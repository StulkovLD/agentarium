# Архитектура agentarium — тур за десять минут

Это рассказ для человека. Истина живёт в `spec/` — здесь только дорога к ней. Каждое решение подано связкой: решение → почему → что было бы без него.

## Идея в одном абзаце

Мультиагентная система здесь **собирается из конфига, а не пишется заново**. Ядро платформы — шина, протокол-конверт, SDK и декларативные маршруты — не знает ни предметной области, ни LLM-фреймворков. Агенты — независимые сервисы в контейнерах, которые не знают друг о друге. Вся связность системы описана одним YAML-файлом.

## Решение 1 — модель шины, как в архитектуре ЭВМ

Модель заимствована у компьютера: общая магистраль, к ней подключаются устройства; устройства друг про друга не знают, протокол подключения един. Мы добавили одно — очередь перед каждым агентом.

**Почему.** Шина — проверенная десятилетиями форма «многие говорят со многими, не зная друг друга». Очередь превращает её в отказоустойчивую: сообщение переживает смерть получателя.

**Без него.** Агенты, зовущие друг друга по HTTP: у каждого — адреса соседей, ретраи, circuit breaker; падение одного каскадит по цепочке; пересборка системы — правка кода. → `spec/00-principles.md`

## Решение 2 — агенты не знают друг о друге; маршруты живут в конфиге

Парсер не вызывает RAG — он даже не знает, что RAG существует. Он публикует факт «заявка разобрана», а кто следующий — решает таблица маршрутов из YAML. Причём отдельного процесса-роутера нет: маршруты становятся биндингами RabbitMQ, доставляет сам брокер.

**Почему.** Одно решение закрывает три требования сразу: гибкость (пересборка = правка YAML), отказоустойчивость (никто ни от кого не зависит в моменте), наблюдаемость (весь трафик в одной точке).

**Без него.** Знание «кто за кем» размазано по коду агентов — «жёсткая реализация одного сценария», ровно то, чем пример быть не должен. → `spec/40-topology.md`

## Решение 3 — шина тупая, интеллект по краям

В шине нет логики. Смысл решают агенты (LLM), связность — декларация, доставка — брокер. Формула: smart endpoints, dumb pipes — выученный индустрией урок после эпохи «умных» Enterprise Service Bus, которые превратились в неподдерживаемых монстров.

**Без него.** Логика стекает в шину, шина становится вторым приложением, менять страшно, тестировать нельзя. → `spec/00-principles.md`

## Решение 4 — SDK как граница расширения

Новый агент — это `class MyAgent(Agent)` с одной функцией `handle()`. Подключение, протокол, ретраи, трейсинг, healthcheck — из SDK. Тип агента объявляет манифест: что принимает, что производит — и рантайм на старте сверяет манифесты с маршрутами.

**Почему.** «Добавление агента любого типа не требует изменения ядра» — выполняется буквально: граница расширения — SDK, ядро никто не открывает. Протокол невозможно нарушить: конверт собирает SDK, а не человек.

**Без него.** Сотни строк одинаковой обвязки в каждом агенте, у каждого автора — чуть по-своему: протокол расходится, наблюдаемость дырявая. → `spec/30-sdk.md`

## Решение 5 — мозги отделены от разъёма

LangGraph и GigaChat живут **внутри агентов PoC** и физически отсутствуют в ядре — это видно по зависимостям контейнеров. Модель — это `provider + base_url + model` в конфиге экземпляра: облако Сбера, свой GPU-сервер или локальная модель подключаются строчкой YAML.

**Почему.** Агент «любого типа» — значит и без LangChain, и вообще без LLM. Платформа не диктует, чем думать.

**Без него.** Платформа превращается в обёртку над конкретным фреймворком и умирает вместе с его мажорным релизом. → `spec/00-principles.md`, `spec/05-stack.md`

## Решение 6 — тип и экземпляр, как класс и объект

Тип агента — код. Экземпляр — тип, поставленный в систему с конфигом. Один тип `rag` стоит в расширенной конфигурации дважды: `knowledge` с коллекцией регламентов и `auditor` с коллекцией инцидентов. Код один, узла два, поведение разное.

**Без него.** Копипаста агентов под каждую вариацию. → `spec/40-topology.md`, `spec/55-poc-scenario.md`

## Решение 7 — fail-fast везде, где можно ошибиться молча

Маршрут в никуда — система не стартует и перечисляет дыры. Коллекция проиндексирована одной моделью эмбеддингов, а агент настроен на другую — отказ на старте с человеческим сообщением (вектора разных моделей несравнимы, поиск вернул бы мусор молча). Ошибка обработки — не тихий фолбэк, а конверт `task.failed`, который маршрутизируется и виден в трейсе.

**Почему.** Тихая деградация хуже остановки: мусор, похожий на результат, разрушает доверие к системе.

**Без него.** «Работает, но почему-то плохо» — худший баг распределённых систем. → `spec/45-storage.md`, `spec/50-observability.md`

## Рассмотренные альтернативы

Коротко: n8n/Temporal/Airflow — центральные движки, шаги не являются независимыми сервисами; Kafka — конвейер для миллионов событий, а не команды малого числа агентов; Redis Streams — транспорт без гарантий брокера, дописывать руками; NATS — экзотика для банковского контура; Chroma — библиотека, а общее хранилище обязано быть сервисом. Полная таблица с причинами — `spec/05-stack.md`.

## Куда это растёт

- **Распределённость по машинам:** шина сетевая — агенты могут жить на разных хостах без изменения платформы; модели — тем более (base_url).
- **MCP:** инструменты агентов через Model Context Protocol — естественный следующий класс интеграции для executor (транспорты stdio / streamable HTTP).
- **Масштабирование вниз:** несколько лёгких агентов в одном процессе-хосте, если контейнер на агента станет дорог.
