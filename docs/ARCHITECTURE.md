# Архитектура agentarium

Обзор ключевых решений платформы. Полные контракты — в `spec/`, здесь только дорога к ним. Каждое решение подано связкой: **решение → почему → что было бы без него.**

## Идея

Мультиагентная система здесь собирается из конфигурации, а не пишется заново под каждый сценарий. Ядро платформы — шина сообщений, протокол, SDK и маршрутизация — не знает ни предметной области, ни LLM-фреймворков. Агенты — независимые сервисы в контейнерах, каждый не знает о существовании остальных. Вся связность системы описана одним YAML-файлом (в спеках он называется топологией).

## Решение 1 — шина сообщений с очередью перед каждым агентом

Общая шина, к которой подключаются агенты; агенты не знают друг о друге, протокол подключения единый. Перед каждым агентом стоит очередь.

**Почему.** Шина — проверенная форма связи «многие со многими, не зная друг друга». Очередь делает её отказоустойчивой: сообщение переживает смерть получателя и дожидается рестарта.

**Без него.** Агенты вызывали бы друг друга напрямую по HTTP. У каждого — адреса соседей, ретраи, circuit breaker; падение одного каскадит по цепочке; пересборка системы требует правки кода. → `spec/00-principles.md`

## Решение 2 — агенты не знают друг о друге, маршруты живут в конфигурации

Parser не вызывает RAG и даже не знает, что RAG существует. Он публикует факт «заявка разобрана», а кто следующий — решает таблица маршрутов в YAML.

Отдельного процесса-маршрутизатора при этом нет: маршруты становятся биндингами RabbitMQ, доставку выполняет сам брокер.

**Почему.** Одно решение закрывает три требования сразу:

- гибкость — пересборка системы это правка YAML;
- отказоустойчивость — никто ни от кого не зависит в моменте;
- наблюдаемость — весь трафик проходит через одну точку.

**Без него.** Знание «кто за кем идёт» размазано по коду агентов. Это жёстко зашитый один сценарий — ровно то, чем платформа быть не должна. → `spec/40-topology.md`

## Решение 3 — логика в агентах, не в шине

В шине нет предметной логики. Смысл решают агенты (LLM), связность задаёт конфигурация, доставку выполняет брокер. Это принцип *smart endpoints, dumb pipes* — урок индустрии после эпохи «умных» Enterprise Service Bus, которые обрастали логикой и становились неподдерживаемыми.

**Без него.** Логика стекает в шину, шина превращается во второе приложение: менять страшно, тестировать нельзя. → `spec/00-principles.md`

## Решение 4 — SDK как граница расширения

Новый агент — это `class MyAgent(Agent)` с одним методом `handle()`. Подключение к шине, протокол, ретраи, трейсинг, healthcheck — всё берётся из SDK. Тип агента объявляет манифест (что принимает, что производит), и рантайм на старте сверяет манифесты с маршрутами.

**Почему.** Требование «добавление агента любого типа не требует изменения ядра» выполняется буквально: граница расширения — SDK, ядро при этом не открывают. Протокол нельзя нарушить случайно — сообщение собирает SDK, а не автор агента.

**Без него.** Сотни строк одинаковой обвязки в каждом агенте, у каждого автора чуть по-своему. Протокол расходится, наблюдаемость дырявая. → `spec/30-sdk.md`

## Решение 5 — модель это данные конфигурации, не зависимость ядра

LangGraph и GigaChat живут внутри агентов PoC и физически отсутствуют в ядре — это видно по списку зависимостей контейнеров. Модель агента задаётся полями конфигурации экземпляра (`provider + base_url + model`), которые платформа передаёт агенту, не разбирая. Поддержка конкретного провайдера — это адаптер внутри агента (PoC реализует GigaChat), а не свойство ядра.

**Почему.** «Агент любого типа» — значит и без LangChain, и вообще без LLM. Платформа не диктует, чем агент думает.

**Без него.** Платформа превращается в обёртку над конкретным фреймворком и устаревает вместе с его мажорным релизом. → `spec/00-principles.md`, `spec/05-stack.md`

**Как это выглядит на PoC.** Модель у каждого агента своя и выбрана по весу задачи:

- parser (классификация заявки) и knowledge (сборка запроса) — лёгкая GigaChat-2;
- executor (планирование и структурированный выбор проверок) — GigaChat-3-Ultra: слабые модели ненадёжно заполняют вложенный список проверок.

Эмбеддинги для RAG по умолчанию идут по API GigaChat (у корпоративного ключа доступны из коробки). Если доступа к эмбеддингам нет, конфигурация `dba-base-local` переключает провайдера на локальную bge-m3 в Ollama-контейнере — платформа обращается к ней по HTTP-адресу из конфигурации, как к облаку. Смена провайдера — одна строка YAML плюс переиндексация, без правок кода. → `spec/diagrams/model-routing.mmd`

## Решение 6 — тип и экземпляр как класс и объект

Тип агента — это код. Экземпляр — тип, поставленный в систему с конфигурацией. Один тип `rag` может стоять в системе дважды с разными коллекциями знаний: код один, узла два, поведение разное (пример конфигурации — в `docs/GUIDE.md`).

Расширенная конфигурация PoC показывает соседнюю грань: новый **тип** `auditor` подключается небольшим модулем через SDK и строкой в каталоге, а маршруты и финал переключаются в конфигурации. Ядро и существующие агенты не тронуты.

**Без него.** Копипаста агентов под каждую вариацию. → `spec/40-topology.md`, `spec/55-poc-scenario.md`

## Решение 7 — fail-fast там, где можно ошибиться молча

- Маршрут ведёт в никуда — система не стартует и перечисляет все дыры сразу.
- Коллекция проиндексирована одной моделью эмбеддингов, а агент настроен на другую — отказ на старте с понятным сообщением. Векторы разных моделей несравнимы, иначе поиск молча возвращал бы мусор.
- Ошибка обработки — не тихий фолбэк, а сообщение `task.failed`, которое маршрутизируется и видно в трейсе.

**Почему.** Тихая деградация хуже остановки: результат, похожий на правильный, но неверный, разрушает доверие к системе.

**Без него.** «Работает, но почему-то плохо» — худший класс багов в распределённых системах. → `spec/45-storage.md`, `spec/50-observability.md`

## Решение 8 — отказ как штатный сценарий

Живучесть — не оговорка в конце, а спроектированный маршрут:

- quorum-очереди с лимитом доставок;
- отдельный dead-letter-exchange с at-least-once стратегией;
- publisher confirms с флагом `mandatory` (confirm без mandatory не заметил бы сообщение, улетевшее мимо биндингов);
- ack входящего сообщения строго после подтверждения исходящего;
- консюмер финалов при упавшей БД держит сообщение, а не сжигает лимит доставок.

Путь сообщения при аварии нарисован в `spec/diagrams/failure-path.mmd`, механика — в `spec/40` и `spec/50`. Механизм закрыт интеграционными тестами на живом брокере (`tests/integration/test_sdk.py`): сообщение переживает смерть консюмера, ядовитое сообщение после трёх доставок уходит в dlq, вотчдог будит зависший `handle`. Скрипт `demo/chaos.sh` показывает это наглядно — убивает агента посреди обработки (ждёт `unacked=1` через Management API).

**Без него.** «Отказоустойчивость» как строчка в README, которая рассыпается на первом же `docker kill`. → `spec/50-observability.md`

## Безопасность

**Защищено по построению:**

- LLM исполнителя **не пишет SQL** — только выбор из allowlist типизированных read-only проверок. Prompt-инъекция в тексте заявки упирается в отсутствие свободного SQL как класса.
- Доступ к целевой БД — отдельная роль с `pg_monitor`, `statement_timeout` и READ ONLY-транзакциями.
- Секреты — только через `.env`, в git их нет. TLS к GigaChat — с цепочкой сертификатов НУЦ Минцифры в образах, а не с отключённой проверкой.
- Недоверенные данные (текст заявки, фрагменты знаний) валидируются схемами на каждой границе.

**Осознанно за скоупом демо:**

- аутентификация шлюза — это локальный инструмент, не публичный сервис;
- идемпотентность `POST /requests` при клиентском ретрае — повторный POST создаёт новую заявку;
- шифрование трафика внутри compose-сети.

Что из этого добавляется при выходе в прод — в разделе ниже.

## Путь к прод-контуру

Что добавляется при выходе из демо в промышленную эксплуатацию, в порядке приоритета:

1. **Аутентификация и авторизация шлюза** — корпоративный SSO/OAuth2 на `POST /requests`, RBAC по типам заявок (кто может просить обновление ОС, а кто только чтение доступов).
2. **Секреты в секрет-менеджер** (Vault или секреты оркестратора) вместо `.env`; ротация ключей LLM-провайдера без перезапуска — конфигурация читается на старте агента, точка внедрения готова.
3. **mTLS между сервисами** — шина, хранилища и агенты в разных сетевых зонах требуют шифрования в транзите. У RabbitMQ и Postgres это конфигурация, не код.
4. **Аудит-лог заявок** — «кто, что, когда просил и что система сделала» в отдельном неизменяемом хранилище. Половина уже есть: каждый шаг в трейсе, финал в Postgres — остаётся вынести в append-only.
5. **Скан образов и SBOM в CI** (trivy/grype) — пайплайн уже двухъярусный, шаг добавляется строкой.
6. **Расширение allowlist на запись** — когда агентам доверят не только проверки, но и изменения: те же типизированные инструменты плюс обязательное подтверждение плана человеком (human-in-the-loop).

Ни один пункт не требует перестройки архитектуры. Границы (SDK, конфигурация, allowlist, финалы) спроектированы так, что прод-контур наращивается на них.

## Рассмотренные альтернативы

Коротко, полная таблица с причинами — в `spec/05-stack.md`:

- **n8n / Temporal / Airflow** — центральные движки; их шаги не являются независимыми сервисами.
- **Kafka** — конвейер для миллионов событий, а не команда из небольшого числа агентов.
- **Redis Streams** — транспорт без гарантий брокера, их пришлось бы дописывать руками.
- **NATS** — редкость для банковского контура.
- **Chroma** — библиотека, а общее хранилище должно быть сервисом.

## Куда это растёт

- **Масштабирование вверх:** несколько реплик одного экземпляра на общей очереди (competing consumers) — брокер сам раздаёт сообщения. Платформа получает это из выбранной модели доставки.
- **Парк целевых БД:** в PoC инсталляция одна; для многих — реестр «логическое имя → DSN» как ещё одно общее хранилище, агенты уже умеют читать хранилища через SDK.
- **Распределённость по машинам:** шина сетевая, агенты могут жить на разных хостах без изменений в платформе; модели — тем более (адрес в `base_url`).
- **Уживчивость с окружением:** платформа наружу не занимает дефолтные порты сервисов (Postgres → 5433, Ollama → 11534), чтобы не конфликтовать с тем, что уже слушает на 5432/11434.
- **MCP:** инструменты агентов через Model Context Protocol — естественный следующий класс интеграции для executor (транспорты stdio / streamable HTTP).
- **Масштабирование вниз:** несколько лёгких агентов в одном процессе-хосте, если контейнер на агента станет дорог.
