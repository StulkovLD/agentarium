# Архитектура agentarium — тур за десять минут

Это рассказ для человека. Истина живёт в `spec/` — здесь только дорога к ней. Каждое решение подано связкой: решение → почему → что было бы без него.

## Идея в одном абзаце

Мультиагентная система здесь **собирается из конфига, а не пишется заново**. Ядро платформы — шина, протокол-конверт, SDK и декларативные маршруты — не знает ни предметной области, ни LLM-фреймворков. Агенты — независимые сервисы в контейнерах, которые не знают друг о друге. Вся связность системы описана одним YAML-файлом.

## Решение 1 — модель шины, как в архитектуре ЭВМ

Модель заимствована у компьютера: общая магистраль, к ней подключаются устройства; устройства друг про друга не знают, протокол подключения един. Мы добавили одно — очередь перед каждым агентом.

**Почему.** Шина — проверенная десятилетиями форма «многие говорят со многими, не зная друг друга». Очередь превращает её в отказоустойчивую: сообщение переживает смерть получателя.

**Без него.** Агенты, зовущие друг друга по HTTP: у каждого — адреса соседей, ретраи, circuit breaker; падение одного каскадит по цепочке; пересборка системы — правка кода. → `spec/00-principles.md`

## Решение 2 — агенты не знают друг о друге; маршруты живут в конфиге

Парсер не вызывает RAG — он даже не знает, что RAG существует. Он публикует факт «заявка разобрана», а кто следующий — решает таблица маршрутов из YAML. Причём отдельного процесса-роутера нет: маршруты становятся биндингами RabbitMQ, доставляет сам брокер.

**Почему.** Одно решение закрывает три требования сразу: гибкость (пересборка = правка YAML), отказоустойчивость (никто ни от кого не зависит в моменте), наблюдаемость (весь трафик в одной точке).

**Без него.** Знание «кто за кем» размазано по коду агентов — «жёсткая реализация одного сценария», ровно то, чем пример быть не должен. → `spec/40-topology.md`

## Решение 3 — шина тупая, интеллект по краям

В шине нет логики. Смысл решают агенты (LLM), связность — декларация, доставка — брокер. Формула: smart endpoints, dumb pipes — выученный индустрией урок после эпохи «умных» Enterprise Service Bus, которые превратились в неподдерживаемых монстров.

**Без него.** Логика стекает в шину, шина становится вторым приложением, менять страшно, тестировать нельзя. → `spec/00-principles.md`

## Решение 4 — SDK как граница расширения

Новый агент — это `class MyAgent(Agent)` с одной функцией `handle()`. Подключение, протокол, ретраи, трейсинг, healthcheck — из SDK. Тип агента объявляет манифест: что принимает, что производит — и рантайм на старте сверяет манифесты с маршрутами.

**Почему.** «Добавление агента любого типа не требует изменения ядра» — выполняется буквально: граница расширения — SDK, ядро никто не открывает. Протокол невозможно нарушить: конверт собирает SDK, а не человек.

**Без него.** Сотни строк одинаковой обвязки в каждом агенте, у каждого автора — чуть по-своему: протокол расходится, наблюдаемость дырявая. → `spec/30-sdk.md`

## Решение 5 — мозги отделены от разъёма

LangGraph и GigaChat живут **внутри агентов PoC** и физически отсутствуют в ядре — это видно по зависимостям контейнеров. Модель — данные конфига экземпляра (`provider + base_url + model`), которые платформа передаёт агенту не разглядывая; поддержка конкретного провайдера — адаптер в мозгах агента (PoC реализует GigaChat), а не свойство ядра.

**Почему.** Агент «любого типа» — значит и без LangChain, и вообще без LLM. Платформа не диктует, чем думать.

**Без него.** Платформа превращается в обёртку над конкретным фреймворком и умирает вместе с его мажорным релизом. → `spec/00-principles.md`, `spec/05-stack.md`

**Так это выглядит на живом PoC.** Модель у каждого агента своя, и выбрана по весу задачи: parser (классификация заявки) и knowledge (сборка запроса) — на лёгкой GigaChat-2; executor (планирование + структурированный выбор проверок) — на GigaChat-3-Ultra, потому что слабые модели не заполняют вложенный список проверок надёжно. Эмбеддинги RAG вынесены на локальную bge-m3 в Ollama-контейнере — платформа ходит к ней по HTTP-адресу из чертежа, ровно как к облаку. Всё это — строки YAML, ни одной правки кода: `spec/diagrams/model-routing.mmd`.

## Решение 6 — тип и экземпляр, как класс и объект

Тип агента — код. Экземпляр — тип, поставленный в систему с конфигом: один тип `rag` может стоять в системе дважды с разными коллекциями — код один, узла два, поведение разное (пример чертежа — в `docs/GUIDE.md`). А расширенная конфигурация PoC показывает соседнюю грань: новый **тип** `auditor` подключается страницей кода через SDK и строкой каталога, а маршруты и финал переключаются чертежом — ядро и существующие агенты не тронуты.

**Без него.** Копипаста агентов под каждую вариацию. → `spec/40-topology.md`, `spec/55-poc-scenario.md`

## Решение 7 — fail-fast везде, где можно ошибиться молча

Маршрут в никуда — система не стартует и перечисляет дыры. Коллекция проиндексирована одной моделью эмбеддингов, а агент настроен на другую — отказ на старте с человеческим сообщением (вектора разных моделей несравнимы, поиск вернул бы мусор молча). Ошибка обработки — не тихий фолбэк, а конверт `task.failed`, который маршрутизируется и виден в трейсе.

**Почему.** Тихая деградация хуже остановки: мусор, похожий на результат, разрушает доверие к системе.

**Без него.** «Работает, но почему-то плохо» — худший баг распределённых систем. → `spec/45-storage.md`, `spec/50-observability.md`

## Решение 8 — отказ как первоклассный сценарий

Живучесть здесь не оговорка в конце, а спроектированный маршрут: quorum-очереди с лимитом доставок, отдельный fanout-DLX с at-least-once стратегией, publisher confirms с `mandatory` (confirm без mandatory не заметил бы конверт, улетевший мимо биндингов), ack входа строго после подтверждения исходящего, консюмер финалов, который при упавшей БД держит конверт, а не сжигает лимит доставок. Путь конверта при аварии нарисован: `spec/diagrams/failure-path.mmd`; механика — `spec/40`, `spec/50`. Механизм закрыт интеграционными тестами на живом брокере (`tests/integration/test_sdk.py`): конверт переживает смерть консюмера, ядовитый конверт после трёх доставок уходит в dlq, вотчдог будит зависший `handle`. `demo/chaos.sh` показывает это глазами — убивает агента ровно посреди обработки (ждёт `unacked=1` по Management API).

**Без него.** «Отказоустойчивость» как строка в README, которая рассыпается на первом же `docker kill`. → `spec/50-observability.md`

## Безопасность — позиция целиком

Что защищено по построению: LLM исполнителя **не пишет SQL** — только выбор из allowlist типизированных read-only проверок (prompt-инъекция в тексте заявки упирается в отсутствие свободного SQL как класса); доступ к целевой БД — отдельная роль с `pg_monitor`, `statement_timeout`, READ ONLY-транзакциями; секреты — только через `.env` (в git не существуют), TLS к GigaChat — цепочка НУЦ Минцифры в образах, не отключение проверки; недоверенные данные (текст заявки, фрагменты знаний) валидируются схемами на каждой границе.

Что осознанно за скоупом PoC: аутентификация шлюза (демо-инструмент инженера, не публичный сервис), идемпотентность `POST /requests` при клиентском ретрае (повторный POST — новая заявка), шифрование трафика внутри compose-сети. Позиция названа, чтобы на защите это был ответ, а не пробел.

**Путь к прод-контуру** — что добавляется при выходе из демо в промышленную эксплуатацию банка, в порядке приоритета:

1. **Аутентификация и авторизация шлюза** — корпоративный SSO/OAuth2 на `POST /requests`, RBAC по типам заявок (кто может просить обновление ОС, а кто только чтение доступов).
2. **Секреты — в секрет-менеджер** (Vault / секреты оркестратора) вместо `.env`; ротация ключей LLM-провайдера без перезапуска — конфиг уже читается на старте агента, точка внедрения готова.
3. **mTLS между сервисами** — шина, склады и агенты в разных сетевых зонах банка требуют шифрования в транзите; у RabbitMQ и Postgres это конфигурация, не код.
4. **Аудит-лог заявок** — «кто, что, когда просил и что система сделала» отдельным неизменяемым хранилищем; половина уже есть: каждый шаг в трейсе, финал в Postgres — остаётся вынести в append-only.
5. **Скан образов и SBOM в CI** (trivy/grype) — пайплайн уже двухъярусный, шаг добавляется строкой.
6. **Расширение allowlist-модели на запись** — когда агентам доверят не только проверки, но и изменения: те же типизированные инструменты + обязательное человеческое подтверждение плана (human-in-the-loop гейт уже напрашивается из финалов чертежа).

Ни один пункт не требует перестройки архитектуры — границы (SDK, чертёж, allowlist, финалы) спроектированы так, что прод-контур наращивается на них, а не вместо них.

## Рассмотренные альтернативы

Коротко: n8n/Temporal/Airflow — центральные движки, шаги не являются независимыми сервисами; Kafka — конвейер для миллионов событий, а не команды малого числа агентов; Redis Streams — транспорт без гарантий брокера, дописывать руками; NATS — экзотика для банковского контура; Chroma — библиотека, а общее хранилище обязано быть сервисом. Полная таблица с причинами — `spec/05-stack.md`.

## Куда это растёт

- **Масштабирование вверх:** реплики одного экземпляра на общей очереди (competing consumers) — брокер сам раздаёт конверты; платформа получает это бесплатно из выбранной модели доставки.
- **Парк целевых БД:** в PoC инсталляция одна; для тысяч — реестр «логическое имя → DSN» как ещё одно общее хранилище, агенты его уже умеют читать через SDK.
- **Распределённость по машинам:** шина сетевая — агенты могут жить на разных хостах без изменения платформы; модели — тем более (base_url).
- **Уживчивость с окружением:** платформа наружу не занимает дефолтные порты сервисов (Postgres → 5433, Ollama → 11534) — у разработчика на 5432/11434 часто уже слушает своё, и клон не должен драться за них.
- **MCP:** инструменты агентов через Model Context Protocol — естественный следующий класс интеграции для executor (транспорты stdio / streamable HTTP).
- **Масштабирование вниз:** несколько лёгких агентов в одном процессе-хосте, если контейнер на агента станет дорог.
