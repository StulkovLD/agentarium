# 40 · Чертёж системы

```
Статус: draft
Владелец: контракт конфигурации — каталог типов, реестр экземпляров, маршруты, вход и финалы
Зависит от: 00, 20, 30
```

## Определение

**Опр.** Чертёж (topology) — YAML-файл, единственный источник состава системы: какие экземпляры агентов существуют, как связаны маршрутами, где вход и что считается финалом. Ядро превращает чертёж в работающую систему.

**Опр.** Каталог типов (catalog) — `agents/catalog.yaml`: декларация каждого типа агента для ядра. Ядро не может импортировать код агентов (они в других образах) — каталог и есть их машиночитаемое лицо.

## Каталог типов

```yaml
# agents/catalog.yaml — владелец знания «какие типы существуют и каковы их контракты»
parser:
  build: agents/parser
  consumes: [request.new]
  produces: [request.parsed, request.rejected]
  config_schema: { model: llm }            # какие поля конфига обязан дать чертёж
rag:
  build: agents/rag
  consumes: [request.parsed]
  produces: [knowledge.found]
  config_schema: { model: llm, collection: str }
executor:
  build: agents/executor
  consumes: [knowledge.found]
  produces: [plan.ready]
  config_schema: { model: llm, target_db: dsn }
auditor:                                   # добавляется вместе с конфигурацией B (слайс S8) —
  build: agents/auditor                    # в базовой поставке этой строки и папки ещё нет
  consumes: [plan.ready]
  produces: [audit.done]
  config_schema: { model: llm, collection: str }
```

Разделение владений: **универсальные** проверки (маршруты, манифесты, схема чертежа) — ядро; **типовые** схемы конфига (`collection`, `target_db`) — каталог. Ядро про «collection» не знает — оно лишь применяет схему типа из каталога. Так требование «добавление агента любого типа без изменения ядра» выполняется и для валидации: новый тип приносит свою строку каталога.

## Чертёж

```yaml
# configs/dba-base.yaml
system: dba-requests

entry: request.new              # тип, который шлюз публикует на POST /requests

agents:                         # реестр экземпляров: имя → тип из каталога + конфиг по схеме типа
  parser:
    type: parser
    model: { provider: gigachat, name: GigaChat-2 }
  knowledge:
    type: rag
    collection: regulations
    model: { provider: gigachat, name: GigaChat-2 }
  executor:
    type: executor
    target_db: env:TARGET_DB_DSN
    model: { provider: gigachat, name: GigaChat-2-Max }

collections:                    # общие коллекции знаний: владелец паспорта эмбеддингов (45)
  regulations:
    source: knowledge/regulations
    embeddings: { provider: gigachat, model: Embeddings }

routes:                         # маршруты: тип конверта → очереди экземпляров
  request.new:      [parser]
  request.parsed:   [knowledge]
  knowledge.found:  [executor]
  plan.ready:       [gateway]
  request.rejected: [gateway]
  task.failed:      [gateway]

finals:                         # финалы: как шлюзу завершать заявку — предметка не в ядре, а в чертеже
  plan.ready:       complete
  request.rejected: fail
  task.failed:      fail
```

Правила:
- **Тип ≠ экземпляр.** Один тип может стоять в системе много раз с разными конфигами: два `rag` с разными коллекциями — два узла из одного кода (класс/объект).
- `env:ИМЯ` подставляется из окружения — секреты и адреса в чертёж не вписываются.
- Fan-out: несколько получателей у типа — конверт получает каждый.
- `gateway` — зарезервированное имя входа/выхода: в `agents` не объявляется, в `routes` использоваться может; валидация знает об этом исключении.
- `finals` — карта «тип → complete | fail». Финальность конфигурируема: в расширенной конфигурации `plan.ready` перестаёт быть финалом и уходит аудитору, финалом становится `audit.done` (см. 55). Шлюз универсален: он исполняет `finals`, не зная предметных имён.

## Шлюз: HTTP-контракт (универсальный, предметки не содержит)

| Запрос | Ответ |
|---|---|
| `POST /requests {"text": "..."}` | `202 {"trace_id": "..."}` — порядок внутри: подтверждённая публикация конверта `entry` → запись `accepted` → ответ; публикация не удалась → `503`, заявка не рождается |
| `GET /requests/{trace_id}` | `200 {"status": "accepted \| done \| failed", "result": {...} \| null}` |
| `GET /requests/{trace_id}` (нет такой) | `404` |

Финал-консюмер пишет результат UPSERT-ом — если шлюз упал в микроокне между публикацией и записью, финал сам создаст строку заявки: вечно висящих «опубликовано, но не записано» не бывает. (Транзакционный outbox — осознанно за скоупом: на этом масштабе он лечит то, что UPSERT закрывает одной строкой.)

Шлюз идемпотентен по финалам правилом «первый выигрывает»: первый финальный конверт фиксирует `status` и `result` заявки, повторные финалы того же `trace_id` игнорируются с записью в лог. Обещать «повторная обработка даст тот же текст» нельзя — в агентах LLM; поэтому идемпотентность живёт на уровне финала, а не содержимого.

## Как чертёж становится системой

Два инструмента с разными владениями: `make gen` работает только с файлами (валидация + генерация compose), `topology apply` — единственный, кто разговаривает с живым брокером. Порядок внутри `make up`: gen → инфраструктура up → **topology apply** → seed → агенты. Объявлять AMQP-объекты до старта RabbitMQ некому и не в чем — потому apply стоит после wait-healthy инфраструктуры.

1. **Маршруты = биндинги брокера.** Отдельного процесса-роутера нет: `topology apply` объявляет topic-exchange `agentarium`, очередь `agentarium.<экземпляр>` каждому агенту, очередь `agentarium.gateway` шлюзу и привязки «тип → очередь» по `routes`. Доставляет сам RabbitMQ. Topic-exchange (а не direct) — ради wildcard-привязок служебных стоков: `*.failed → agentarium.dlq`. **Владелец объявления — только apply**; SDK агента делает passive-проверку своей очереди — только факт существования (AMQP при passive-declare аргументы не сверяет) — и не объявляет ничего: идентичность аргументов гарантируется тем, что объявляет их ровно один владелец, а контрольная сверка аргументов — забота smoke-проверки через Management API.
2. **Мёртвые письма.** Отдельный dead-letter-exchange `agentarium.dlx` (fanout — ключ маршрутизации мёртвого конверта неважен, ловится любой) с привязанной очередью `agentarium.dlq`; каждая quorum-очередь объявляется с `x-dead-letter-exchange=agentarium.dlx`, `x-dead-letter-strategy=at-least-once` и `overflow=reject-publish` — иначе дедлеттеринг у quorum-очередей по умолчанию at-most-once, и «конверт не теряется» было бы неправдой. Дедлеттерить в основной exchange нельзя — конверт вернулся бы своим же ключом в свою же очередь по кругу.
3. **Реконсиляция.** Биндинги приводятся к чертежу как diff (список текущих — через Management API): недостающие объявить, лишние снять. Служебные объекты (dlx, dlq, их привязки, очередь шлюза) — в защищённом списке, diff их не трогает. Страховка S3-гейта: при перерасходе времени — откат на идемпотентное объявление без снятия сирот.
4. **Compose агентов генерируется.** `make gen` эмитит `docker-compose.agents.yml` из `agents` + каталога: build, env (`AGENT_INSTANCE`), mount чертежа, healthcheck на `/health`, `restart: unless-stopped`, `init: true` (PID 1 — tini: без него рабочий процесс становится PID 1 и делается неубиваемым изнутри — а сценарию аварии в демо нужно честно убивать процесс, см. 50). Запуск — всегда явным объединением: `docker compose -f docker-compose.yml -f docker-compose.agents.yml up` (Compose сам второй файл не подключает). Шлюз получает чертёж тем же способом (mount + env с именем конфига); смена `CONFIG` перезапускает и его. Шаблон фиксированный и маленький. Страховка: генератор съедает время — S3-гейт разрешает откат на статичный compose (см. 70).

## Валидация — fail-fast, двумя рубежами

**Рубеж 1 — `make gen` (полный чертёж):** схема YAML цела; каждый `type` есть в каталоге; конфиг экземпляра проходит `config_schema` типа; маршрут не ссылается на несуществующий экземпляр (кроме `gateway`); тип маршрутизирован только тем, чей `consumes` его принимает; каждый `produces` имеет маршрут — конверту есть куда лететь; `entry` замкнут на маршрут, и его payload-схема — `{text: str}` (шлюз принимает текст заявки, других входов у платформы нет); **каждый ключ `finals` маршрутизирован в `gateway`** и в `gateway` ведут только типы из `finals` — двусторонняя проверка; `task.failed` маршрутизирован и присутствует в `finals`; достижимый `complete`-финал ровно один — гонка двух финалов не проходит валидацию; у каждой упомянутой коллекции есть блок в `collections`.

Системные типы (`task.failed` — его эмитит SDK, не агент) — в явном carve-out валидации, как и имя `gateway`.

Примитивы `config_schema`: `str` — непустая строка; `dsn` — строка-DSN, разрешается через `env:`; `llm` — блок `{provider: str, name: str, base_url?: str}`. Больше примитивов нет, пока не понадобятся.

**Рубеж 2 — старт агента (свой фрагмент):** имя экземпляра существует в чертеже; конфиг проходит схему; коллекция сверяется с паспортом (45). 

Полусобранная система не поднимается. Либо чертёж цел, либо громкий отказ со списком всех дыр сразу.
