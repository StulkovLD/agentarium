Зависит от: 00, 20

## Определение

**Опр.** SDK — библиотека `agentarium`, которой агент подключается к платформе. Розетка: прибор любой, разъём один. SDK не знает про LangChain, GigaChat и предметную область — это мозги, они дело агента.

## Весь агент целиком

```python
from agentarium import Agent, Reply

class ParserAgent(Agent):
    consumes = ["request.new"]
    produces = ["request.parsed", "request.rejected"]

    async def handle(self, envelope) -> Reply | None:
        text = envelope.payload["text"]
        parsed = await my_brains(text, self.config)   # мозги любые: LangGraph, голый API, регулярки
        return Reply(type="request.parsed", payload=parsed)
```

Запуск: `python -m agentarium run` — имя экземпляра только из env `AGENT_INSTANCE`, чертёж смонтирован в контейнер (оба параметра эмитит генератор compose, 40). Один канал конфигурации, не два.

## Что SDK делает за агента

| Зона | Что именно |
|---|---|
| Подключение | соединение с шиной, passive-проверка своей quorum-очереди (владелец объявления — `topology apply`, 40), потребление с `prefetch=1` — один конверт в обработке: и семантика «взял — доведи», и детерминизм chaos-сцены (`unacked=1`) стоят на этом |
| Протокол | валидация входного конверта **и его payload по схеме типа**; сборка исходящего: id, trace_id, causation_id, ts, producer — агент их не трогает; валидация `Reply.type` против `produces` и payload против схемы типа |
| Надёжная публикация | durable + persistent + publisher confirms + `mandatory=True` с return-callback как ошибкой — confirm без mandatory не заметил бы конверт, улетевший в exchange без подходящего биндинга; **ack входа только после подтверждённой публикации исходящего** — потерянных конвертов не существует |
| Повторы и ошибки | ретраи 1s/3s/9s — повтор всего `handle` (внутрь мозгов SDK не смотрит); исчерпание → подтверждённый `task.failed`, затем ack; ошибка контракта не ретраится (50) |
| Наблюдаемость | спан на обработку; `traceparent` в AMQP-заголовках туда и обратно; structlog с trace_id (50) |
| Здоровье | HTTP `/health`: «жив» = процесс жив **и** соединение с шиной живо; соединение robust с авто-реконнектом, безнадёжный обрыв → громкий выход; вотчдог: `handle` дольше таймаута (дефолт 120s, переопределяется в каталоге; таймаут > бюджет ретраев + латентность LLM) → громкий выход процесса (docker поднимет) |
| Конфиг | `self.config` — блок экземпляра из чертежа, уже проверенный схемой типа из каталога |
| Склады | клиенты хранилищ — модуль платформы `storage`: `postgres_pool()`, `qdrant(collection, topology)` с паспорт-сверкой (45); агент импортирует их из ядра напрямую |

## Схемы payload — контракт-труба до конца

Каждый тип сообщения владеет Pydantic-схемой payload. Реестр машиночитаем и **собирается, а не редактируется**: базовые типы (`task.failed`) объявляет ядро, а схемы своих `produces` каждый тип агента приносит в собственном пакете — реестр собирает их по каталогу на старте. Добавление нового типа агента не трогает общий пакет контрактов: схема `audit.done` приезжает вместе с auditor, как и весь его код.

Канал доставки схем между изолированными образами — явный: у каждого типа схемы лежат в отдельном лёгком contract-модуле (только Pydantic-модели, без мозгов и их зависимостей), и сборка кладёт contract-модули **всех** типов каталога в каждый образ, включая шлюз. Потребитель валидирует вход по схеме типа-производителя, не имея его кода — только его контракт. SDK валидирует payload на входе до `handle` и на выходе после `Reply` — несовместимые данные ловятся на границе, а не глубиной агента. Изменение схемы несовместимо — новый тип (`request.parsed.v2`), старый живёт до вымирания.

## Контракт `handle`

- Вход: валидный конверт с валидным payload. Невалидное до `handle` не доходит — `task.failed` без ретраев.
- Выход: `Reply(type, payload)` — SDK проверит, завернёт, надёжно опубликует; `None` — конец ветки.
- Манифест (`consumes`/`produces`) — контракт типа; его копия в каталоге (40) — лицо типа для валидации чертежа. Тест каталога сверяет копию с кодом.

## Жизненный цикл

```
старт → env: имя экземпляра → чертёж (mount) → конфиг + схема типа → паспорта коллекций →
→ соединение → петля: конверт → span → handle (под вотчдогом) → publish+confirm → ack →
→ SIGTERM → дообработать текущий конверт → закрыть соединение → выход
```

Изящное завершение: начатый конверт либо доведён и подтверждён, либо не подтверждён и вернётся в очередь.

## Чего SDK не делает — по принципам

- Не создаёт LLM-клиентов и не импортирует LLM-фреймворки: передаёт конфиг модели как данные.
- Не маршрутизирует: тип вернул агент, дальше — чертёж.
- Не прячет ошибки: тихих фолбэков нет.
