# 55 · Сценарий PoC: заявки DBA

```
Статус: draft
Владелец: продуктовый сценарий примера — единственное место, где живёт предметка
Зависит от: 20, 30, 40, 45
```

Платформа не знает про DBA — предметная область целиком живёт в этом файле, в агентах каталога, в `knowledge/` и в чертежах `configs/`. Тема — по рекомендации проверяющего: система обработки заявок по работам администраторов баз данных.

## Сюжет

Инженеру прилетают заявки человеческим языком: «проверь доступы ivanov на проде», «подготовь обновление PostgreSQL на db-billing до 16.3». Система разбирает заявку, достаёт регламент, строит план работ и выполняет реальные проверки против живой целевой БД. Человек получает план и вердикт — рутина типовых заявок снята.

## Состав

**Опр.** Целевая БД (target-db) — PostgreSQL-контейнер-«пациент», который система сопровождает. Наполнен ролями и пользователями для демо. Не путать со складом состояния платформы.

| Узел | Тип | Мозги | Работа |
|---|---|---|---|
| gateway | сервис ядра | нет | вход/выход по HTTP-контракту из 40; исполняет `entry` и `finals` чертежа |
| parser | `parser` | langchain-gigachat, structured output | текст → интент + сущности; не-заявка → отказ |
| knowledge | `rag` | GigaChat Embeddings + GigaChat | интент + сущности → фрагменты регламентов из Qdrant |
| executor | `executor` | **LangGraph** + GigaChat-2-Max | регламент + заявка → план работ + реальные проверки против target-db |
| auditor (конфигурация B) | `auditor` | GigaChat Embeddings + GigaChat | план → сверка с историей инцидентов |

LLM-фреймворки по-честному: LangGraph стоит там, где есть граф (executor: план → проверки → вердикт, с ветвлением); parser и knowledge — прямые вызовы langchain-gigachat, граф из одного узла был бы золочением.

### parser: `request.new → request.parsed | request.rejected`

Интенты: `check_access` · `update_db_version` · `update_os` · `compliance_check`. Сущности: `user, database, host, environment, target_version, deadline` — что найдено. Один structured-вызов; смысл решает LLM, кода-угадывания нет. Текст не является заявкой → `request.rejected {reason}` — по маршруту прямо в шлюз, заявка честно завершается отказом, путь виден в трейсе.

Payload: `{text, intent, entities}` — исходный текст едет дальше: труба несёт весь контекст, downstream-агентам не приходится ничего доставать «откуда-то ещё».

### knowledge: `request.parsed → knowledge.found`

Запрос = интент + сущности → вектор → поиск в коллекции из конфига экземпляра (top-5, порог схожести). Выход-payload: `{request: {text, intent, entities}, chunks: [{text, source, heading}]}` — контекст заявки прокладывается дальше нетронутым. Пусто — честное `chunks: []`, решает следующий агент (один владелец на решение).

### executor: `knowledge.found → plan.ready`

LangGraph-граф: план по регламенту → выбор проверок → исполнение → вердикт. Безопасность — по-взрослому, аудитория ревью это DBA:

- LLM **не пишет SQL**. Он выбирает из allowlist именованных проверок с типизированными аргументами: `pg_version()`, `user_roles(user)`, `user_privileges(user, database)`, `db_size(database)`, `active_connections()`. Исполняет их детерминированный код.
- Подключение к target-db — **отдельная read-only роль** `readonly_executor` с `GRANT pg_monitor` (иначе `active_connections()` через `pg_stat_activity` видела бы только собственные сессии), `statement_timeout` выставлен, транзакции — `READ ONLY`. Весь набор проверок — только чтение, аргументы типизированы схемой инструмента, сырого SQL от LLM не существует.
- Целевая БД в PoC одна: `entities.database` сверяется с ней, маппинг «логическое имя → DSN» для парка инсталляций — честно за скоупом (одна строка в ARCHITECTURE «куда растёт»).
- Граф ветвится по-настоящему: узел вердикта сравнивает результаты проверок с планом — расходятся (версия уже целевая, пользователь не существует, прав больше, чем ожидалось) → переход в узел пересборки плана с фактами, сходятся → финализация. Это та развилка, ради которой executor и держит LangGraph.

Выход: `{plan: [шаги], checks: [{name, args, result}], verdict, sources}`.

### auditor: `plan.ready → audit.done` (только конфигурация B)

Отдельный **тип** из каталога: план → вектор → поиск по коллекции `incidents` → замечания «на этих граблях уже стояли». Выход: `{plan, checks, verdict, sources, audit: {warnings}}` — обогащённый финал.

## База знаний — `knowledge/<коллекция>/`

Раскладка по коллекциям задаёт принадлежность: `knowledge/regulations/*.md` — 5–6 регламентов (обновление минорной/мажорной версии PostgreSQL, выдача и проверка доступов, ролевая модель, compliance-чеклист, бэкапы); `knowledge/incidents/*.md` — 3–4 истории инцидентов для конфигурации B. Markdown с заголовками — по ним режутся чанки. Пишем сами по мотивам публичных практик DBA.

## Две конфигурации — доказательство расширяемости

**A. `configs/dba-base.yaml`** — цепочка: parser → knowledge → executor → gateway. Финал: `plan.ready: complete`.

**B. `configs/dba-extended.yaml`** — добавлен аудит плана. Два изменения:
1. **Новый тип агента** `auditor` — маленький модуль через SDK (страница кода), строка в каталоге. Ядро и существующие агенты **не тронуты** — это дословно требование №3, показанное в лоб: `git diff` между конфигурациями трогает только YAML и папку нового агента.
2. **Маршруты и финал переключены в чертеже:** `plan.ready → [auditor]`, `audit.done → [gateway]`, финал теперь `audit.done: complete`. План не завершает заявку — завершает аудит. Гонки нет: финал в каждой конфигурации ровно один.

Демонстрация «тип ≠ экземпляр» (класс/объект) живёт отдельно и дёшево: `knowledge` и `auditor` — оба ходят в Qdrant с разными коллекциями, а в `docs/GUIDE.md` показан чертёж с двумя экземплярами `rag` в одной системе.

Переключение: `make demo CONFIG=dba-extended` — пересборка без правки ядра.

## Демо-заявки (войдут в `make demo`)

1. «Проверь, какие доступы у пользователя ivanov в базе billing на проде» → check_access, реальные роли из target-db.
2. «Подготовь план обновления PostgreSQL на db-billing с 16.1 до 16.3» → update_db_version, план по регламенту, проверка текущей версии.
3. «Что за ерунда:))» → request.rejected, честный отказ «не заявка», путь виден в трейсе — отказ тоже наблюдаем.
