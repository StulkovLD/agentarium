# 55 · Сценарий PoC: заявки DBA

```
Статус: draft
Владелец: продуктовый сценарий примера — единственное место, где живёт предметка
Зависит от: 20, 30, 40, 45
```

Платформа не знает про DBA — предметная область целиком живёт в этом файле, в агентах каталога, в `knowledge/` и в чертежах `configs/`. Тема выбрана по рекомендации проверяющего: система обработки заявок по работам администраторов баз данных.

## Сюжет

Инженеру прилетают заявки человеческим языком: «проверь доступы ivanov на проде», «подготовь обновление PostgreSQL на db-billing до 16.3». Система разбирает заявку, достаёт регламент, строит план работ и выполняет реальные проверки против живой целевой БД. Человек получает план и вердикт — рутина типовых заявок снята.

## Состав: три типовых агента + шлюз + пациент

**Опр.** Целевая БД (target-db) — отдельный PostgreSQL-контейнер-«пациент», который система сопровождает. Не путать со складом состояния платформы.

| Узел | Тип | Мозги | Работа |
|---|---|---|---|
| gateway | сервис ядра | нет | `POST /requests` → конверт `request.new`, запись заявки; финалы (`plan.ready`, `task.failed`) → результат в Postgres; `GET /requests/{id}` |
| parser | `parser` | LangGraph + GigaChat | текст → интент + сущности |
| knowledge | `rag` | GigaChat Embeddings + GigaChat | интент + сущности → релевантные фрагменты регламентов из Qdrant |
| executor | `executor` | LangGraph + GigaChat-2-Max | регламент + заявка → план работ + реальные проверки против target-db |

### parser: `request.new → request.parsed`

Интенты: `check_access` · `update_db_version` · `update_os` · `compliance_check` · `other`.
Сущности: `user, database, host, environment, target_version, deadline` — что найдено.
LLM-вызов один, выход структурный (function calling). Ноль эвристик: никакого поиска слов кодом.

### knowledge: `request.parsed → knowledge.found`

Запрос = интент + сущности → вектор → поиск в коллекции из конфига экземпляра (top-5, порог схожести). Выход: фрагменты с указанием источника (файл, заголовок). Пустой результат — честное поле `chunks: []`, решает следующий агент, не этот (один владелец на решение).

### executor: `knowledge.found → plan.ready`

LangGraph-граф: план по регламенту → проверки → вердикт.
Проверки — **детерминированный код** с реальными запросами к target-db: `SELECT version()`, роли и права пользователя, размер БД, активные подключения. LLM решает, *какие* проверки уместны и что значит их результат; *выполняет* их код. Выход: `{plan: [шаги], checks: [{name, result}], verdict, sources}`.

## База знаний — `knowledge/`

Пишем сами, 6–8 файлов-регламентов по мотивам публичных практик DBA: обновление минорной/мажорной версии PostgreSQL, выдача и проверка доступов, ролевая модель, compliance-чеклист, регламент бэкапов, действия при инцидентах. Формат — markdown с заголовками (по ним режутся чанки). Коллекция `incidents` — 3–4 файла «историй»: что ломалось и почему, для конфигурации B.

## Две конфигурации — доказательство гибкости

**A. `configs/dba-base.yaml`** — цепочка: `request.new → parser → knowledge → executor → gateway`.

**B. `configs/dba-extended.yaml`** — те же кубики + два изменения только в YAML:
1. Второй экземпляр типа `rag`: `auditor` с коллекцией `incidents` — тип один, экземпляра два, конфиги разные (класс/объект).
2. Fan-out: `plan.ready: [gateway, auditor]`; auditor сверяет план с историей инцидентов → `audit.done → gateway`; шлюз дописывает замечания аудита к результату заявки.

Переключение: `make demo CONFIG=dba-extended` — код не менялся, ядро не менялось, система стала другой.

## Демо-заявки (войдут в `make demo`)

1. «Проверь, какие доступы у пользователя ivanov в базе billing на проде» → check_access, реальные роли из target-db.
2. «Подготовь план обновления PostgreSQL на db-billing с 16.1 до 16.3» → update_db_version, план по регламенту, проверка текущей версии.
3. «Что за ерунда:))» → other, честный ответ «не заявка», путь виден в трейсе — отказ тоже наблюдаем.
