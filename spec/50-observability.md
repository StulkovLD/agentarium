# 50 · Наблюдаемость и живучесть

```
Статус: draft
Владелец: контракт сквозных механизмов — трейсы, логи, здоровье, отказы
Зависит от: 00, 20, 30
```

Оба механизма — свойства платформы, не агентов: живут в SDK и в конфигурации инфраструктуры, достаются любому агенту бесплатно, включая написанные через год.

## Наблюдаемость

Две нити, каждая со своей работой:

- **`trace_id`** — бизнес-корреляция: рождается в шлюзе, копируется в каждый конверт, попадает в каждый лог. «Покажи всё по заявке X».
- **`traceparent` (W3C)** — склейка спанов: SDK кладёт контекст OpenTelemetry в **AMQP-заголовки** при публикации и извлекает при потреблении, открывая связанный consumer-спан. В теле конверта его нет — это транспортный слой, владелец — SDK. Без этого Jaeger показал бы N отдельных трейсов вместо одного водопада. Начало и конец нити — у шлюза: он открывает root-спан заявки на `POST /requests` (business `trace_id` — атрибутом), инжектит traceparent в entry-конверт, а его финал-консюмер довязывает завершение к той же нити.

Дальше просто: спан на каждую обработку (`agent.handle`: producer, type, envelope id, **trace_id**, попытка — по trace_id заявка ищется в Jaeger руками), экспорт по OTLP в Jaeger; логи structlog JSON в stdout с обязательными ключами `ts, level, agent, trace_id, type, event`; шина видна глазами в RabbitMQ management UI.

## Живучесть

Слои защиты, снизу вверх:

| Слой | Механизм | Что закрывает |
|---|---|---|
| Публикация | durable exchange и очереди, `delivery_mode=persistent`, publisher confirms; **ack входа только после подтверждённой публикации исходящего** | Reply не теряется между «сделал» и «отдал»; перезапуск брокера не съедает конверты |
| Доставка | quorum-очереди с `x-delivery-limit=3` → после исчерпания брокер уводит конверт в `agentarium.dlx` → `agentarium.dlq` (отдельный fanout-DLX, см. 40) | ядовитый конверт (роняющий процесс) не зацикливает доставку вечно |
| Обработка | ретраи внутри SDK: 3 повторных запуска `handle` целиком с паузами 1s → 3s → 9s (SDK не видит внутрь мозгов — гранулярность повтора только весь `handle`; повтор LLM-агента платный, это осознанная цена), номер попытки — в спане | дрожание сети, таймаут LLM; конверт неподтверждён на время «паузы ~13s + длительность самих попыток» |
| Исчерпание | SDK публикует `task.failed` (подтверждённо), затем ack входа; `task.failed` летит по маршруту (обычно в шлюз → заявка честно `failed`), а wildcard-биндинг `*.failed` дублирует его в `agentarium.dlq` для разбора | ошибка не тихая и не теряется |
| Тихая смерть | конверт, погибший мимо `task.failed` (краш процесса, delivery-limit), оказывается в `agentarium.dlq`; мини-консюмер dlq в составе шлюза переводит заявку в `failed` по `trace_id` конверта | заявок, навсегда зависших в `accepted`, не существует — тихое зависание запрещено тем же законом, что и тихая деградация |
| Процесс | вотчдог SDK: `handle` дольше таймаута (дефолт 120s, переопределяется полем типа в каталоге; инвариант: таймаут > бюджет ретраев + латентность LLM) → громкий выход процесса → docker `restart: unless-stopped` поднимает; соединение с шиной — robust с авто-реконнектом, безнадёжный обрыв → громкий выход; `/health` отвечает «жив» только при живом соединении с шиной | и умерший, и зависший агент возвращаются в строй (honest-примечание: сам по себе docker unhealthy-контейнер не перезапускает — потому вотчдог обязателен) |
| Система | падение одного агента не трогает остальных: связи только через шину | нет каскадных смертей |

Правила:
- Доставка — **at-least-once**: конверт может прийти повторно, и повторная обработка LLM-агентом может дать другой текст — это нормально. Идемпотентность живёт на уровне финала: «первый выигрывает» у шлюза (40), а не на уровне побайтового содержимого.
- Ретраи — на транспортные и LLM-ошибки (любое исключение из `handle`). Ошибка контракта (невалидный конверт/payload) не ретраится — это баг: `task.failed` с первой попытки, повторение не чинит неправильное.

## Сценарий-доказательство (войдёт в демо)

```
make demo &                                        # заявка в полёте
docker exec agentarium-knowledge-1 kill -9 1       # авария процесса изнутри — PID 1 убит
# конверт request.parsed невозмутимо лежит в очереди, система жива, шлюз отвечает
# restart-политика поднимает агента → конверт дообработан → заявка завершается
```

Тонкость, которую легко провалить: `docker kill` снаружи Docker считает **ручной остановкой**, и `restart: unless-stopped` такой контейнер не поднимет — аварию имитируем убийством PID 1 изнутри, это честный краш процесса, на который рестарт-политика обязана среагировать.
