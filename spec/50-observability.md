# 50 · Наблюдаемость и живучесть

```
Статус: draft
Владелец: контракт сквозных механизмов — трейсы, логи, здоровье, отказы
Зависит от: 00, 20, 30
```

Оба механизма — свойства платформы, не агентов: живут в SDK и в конфигурации инфраструктуры, достаются любому агенту бесплатно, включая написанные через год.

## Наблюдаемость

Две нити, каждая со своей работой:

- **`trace_id`** — бизнес-корреляция: рождается в шлюзе, копируется в каждый конверт, попадает в каждый лог. «Покажи всё по заявке X».
- **`traceparent` (W3C)** — склейка спанов: SDK кладёт контекст OpenTelemetry в **AMQP-заголовки** при публикации и извлекает при потреблении, открывая связанный consumer-спан. В теле конверта его нет — это транспортный слой, владелец — SDK. Без этого Jaeger показал бы N отдельных трейсов вместо одного водопада.

Дальше просто: спан на каждую обработку (`agent.handle`: producer, type, envelope id, попытка), экспорт по OTLP в Jaeger; логи structlog JSON в stdout с обязательными ключами `ts, level, agent, trace_id, type, event`; шина видна глазами в RabbitMQ management UI.

## Живучесть

Слои защиты, снизу вверх:

| Слой | Механизм | Что закрывает |
|---|---|---|
| Публикация | durable exchange и очереди, `delivery_mode=persistent`, publisher confirms; **ack входа только после подтверждённой публикации исходящего** | Reply не теряется между «сделал» и «отдал»; перезапуск брокера не съедает конверты |
| Доставка | quorum-очереди с `x-delivery-limit=3` → после исчерпания брокер сам уводит конверт в DLX → `agentarium.dlq` | ядовитый конверт (роняющий процесс) не зацикливает доставку вечно |
| Обработка | ретраи внутри SDK: 3 попытки с паузами 1s → 3s → 9s, номер попытки — в спане | дрожание сети, таймаут LLM; конверт не висит неподтверждённым дольше ~15s |
| Исчерпание | SDK публикует `task.failed` (подтверждённо), затем ack входа; `task.failed` летит по маршруту (обычно в шлюз → заявка честно `failed`), а wildcard-биндинг `*.failed` дублирует его в `agentarium.dlq` для разбора | ошибка не тихая и не теряется |
| Процесс | вотчдог SDK: `handle` дольше таймаута → громкий выход процесса → docker `restart: unless-stopped` поднимает; `/health` отвечает «жив» только при живом соединении с шиной | и умерший, и зависший агент возвращаются в строй (honest-примечание: сам по себе docker unhealthy-контейнер не перезапускает — потому вотчдог обязателен) |
| Система | падение одного агента не трогает остальных: связи только через шину | нет каскадных смертей |

Правила:
- Доставка — **at-least-once**: конверт может прийти повторно. Финалы у шлюза идемпотентны (40); агенты PoC — вычислительные, повторная обработка даёт эквивалентный результат.
- Ретраи — на транспортные и LLM-ошибки. Ошибка контракта (невалидный конверт/payload) не ретраится — это баг: `task.failed` с первой попытки, повторение не чинит неправильное.

## Сценарий-доказательство (войдёт в демо)

```
make demo &                        # заявка в полёте
docker kill agentarium-knowledge-1 # убили агента посреди обработки
# конверт request.parsed невозмутимо лежит в очереди, система жива, шлюз отвечает
# restart-политика поднимает агента → конверт дообработан → заявка завершается
```
