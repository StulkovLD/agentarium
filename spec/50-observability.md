# 50 · Наблюдаемость и живучесть

```
Статус: draft
Владелец: контракт сквозных механизмов — трейсы, логи, здоровье, отказы
Зависит от: 00, 20, 30
```

Оба механизма — свойства платформы, не агентов: живут в SDK и в конфигурации инфраструктуры, достаются любому агенту бесплатно, включая написанные через год.

## Наблюдаемость

**Нить — `trace_id`.** Рождается в шлюзе вместе с заявкой, копируется в каждый конверт, попадает в каждый спан и каждую строку лога. Один идентификатор — весь путь.

- **Трейсы:** SDK открывает спан на каждую обработку (`agent.handle`, атрибуты: producer, type, envelope id, попытка). Экспорт по OTLP в Jaeger. Путь заявки в UI — водопад: gateway → parser → knowledge → executor.
- **Логи:** structlog, JSON в stdout. Обязательные ключи каждой записи: `ts, level, agent, trace_id, type, event`. Смотреть — `docker compose logs`, фильтровать — `jq`.
- **Шина видна глазами:** RabbitMQ management UI — очереди, глубина, скорость. Отдельно писать нечего — свойство выбранного брокера.

## Живучесть

Слои защиты, снизу вверх:

| Слой | Механизм | Что закрывает |
|---|---|---|
| Конверт | очередь переживает падение получателя: неподтверждённый конверт возвращается | смерть агента посреди работы — конверт не потерян |
| Обработка | ретраи в SDK: 3 попытки с паузами 1s → 5s → 25s | дрожание сети, таймаут LLM |
| Исчерпание | конверт `task.failed` по маршруту (обычно в шлюз) + копия в очередь `agentarium.dlq` | ошибка не тихая: заявка честно завершается, конверт доступен разбору |
| Процесс | docker healthcheck по `/health` + `restart: unless-stopped` | зависший или умерший контейнер поднимается сам |
| Система | падение одного агента не трогает остальных: связи только через шину | нет каскадных смертей |

Правила:
- Очереди durable, подтверждение — только после успешного `handle`. «Взял и умер» = конверт вернулся.
- Ретраи — на транспортные и LLM-ошибки. На ошибку контракта (невалидный конверт) ретраев нет — это баг, он летит в `task.failed` с первой попытки: повторение не чинит неправильное.
- Пуассоновских чудес не ждём: сообщение может прийти повторно (at-least-once). Обработка обязана быть идемпотентной; для PoC это дёшево — шаги вычислительные, финальная запись результата перезаписывает то же поле той же заявки.

## Сценарий-доказательство (войдёт в демо)

```
make demo &            # заявка в полёте
docker kill agentarium-knowledge-1
# наблюдаем: конверт request.parsed лежит в очереди, система жива, шлюз отвечает
# docker restart поднимает агента → конверт дообработан → заявка завершается
```
