# 45 · Общие склады

```
Статус: draft
Владелец: контракт общих хранилищ — состояние и знания
Зависит от: 00, 30
```

Два склада, два разных вопроса: Postgres отвечает «что происходит с заявками», Qdrant — «что система знает». Агент получает доступ только через SDK (`self.storage`) — там живут проверки.

## PostgreSQL — склад состояния

Единственная таблица бизнес-состояния:

```sql
CREATE TABLE requests (
    trace_id    uuid PRIMARY KEY,
    text        text        NOT NULL,          -- заявка как её подал человек
    status      text        NOT NULL,          -- accepted | in_progress | done | failed
    result      jsonb,                         -- итоговый план / причина ошибки
    created_at  timestamptz NOT NULL DEFAULT now(),
    updated_at  timestamptz NOT NULL DEFAULT now()
);
```

- Пишут: шлюз (рождение заявки и финал), никто больше. Промежуточные шаги — не состояние, а наблюдаемость: они живут в трейсе, не в базе.
- Читают: шлюз (`GET /requests/{trace_id}`), агенты — по потребности через SDK.
- Миграция — один SQL-файл, применяется на старте. Инструмент миграций не заводим: одна таблица его не оправдывает (отсутствие лишнего).

## Qdrant — склад знаний

**Опр.** Коллекция — именованное векторное пространство с паспортом. Пример: `regulations` (регламенты DBA), `incidents` (история инцидентов).

### Паспорт коллекции

Вектора разных моделей живут в разных пространствах: проиндексировал одной моделью, ищешь другой — получишь мусор, причём молча. Поэтому при создании коллекции платформа записывает в её метаданные паспорт:

```json
{ "embed_provider": "gigachat", "embed_model": "Embeddings", "dim": 1024, "created_by": "agentarium ingest" }
```

**Fail-fast:** `self.storage.qdrant(collection)` сверяет модель эмбеддингов из конфига агента с паспортом коллекции **на старте агента**. Не совпало — отказ с человеческим сообщением: *«коллекция regulations проиндексирована Embeddings (1024), в конфиге агента bge-m3 — смени модель или переиндексируй»*. Смена модели эмбеддингов = переиндексация, других путей нет.

### Инжест знаний

`make seed` — утилита платформы (`tools/ingest`):

```
knowledge/*.md → чанки (по заголовкам, ~500 токенов, перехлёст 50)
             → вектора (модель из паспорта коллекции)
             → Qdrant c payload: {source: имя файла, heading, text}
```

Инжест идемпотентен: повторный `make seed` пересоздаёт коллекцию целиком. Содержимое `knowledge/` — предметка PoC (`55-poc-scenario.md`), сама утилита предметки не знает.

## Чего на складах нет — по принципам

- Конверты в базах не хранятся: шина — не склад, склад — не шина.
- Кэшей «для скорости» нет: не проблема этого масштаба (лишнее).
- Доступ агентов к складам мимо SDK запрещён — проверки паспорта живут в одном месте.
