# 45 · Общие склады

```
Статус: draft
Владелец: контракт общих хранилищ — состояние и знания
Зависит от: 00, 30, 40
```

Два склада, два разных вопроса: Postgres отвечает «что происходит с заявками», Qdrant — «что система знает». Агент получает доступ только через SDK (`self.storage`) — там живут проверки.

## PostgreSQL — склад состояния

```sql
CREATE TABLE requests (
    trace_id    uuid PRIMARY KEY,
    text        text        NOT NULL,          -- заявка как её подал человек
    status      text        NOT NULL,          -- accepted | done | failed
    result      jsonb,                         -- итоговый план / причина отказа
    created_at  timestamptz NOT NULL DEFAULT now(),
    updated_at  timestamptz NOT NULL DEFAULT now()
);
```

- Пишет только шлюз: `accepted` при рождении, `done`/`failed` по `finals` чертежа. Промежуточных статусов нет — путь заявки это наблюдаемость (трейс), не состояние.
- Финалы идемпотентны: повторная доставка финального конверта перезаписывает `result` тем же содержимым.
- Миграция — один SQL-файл, применяется на старте. Инструмент миграций не заводим: одна таблица его не оправдывает.

## Qdrant — склад знаний

**Опр.** Коллекция — именованное векторное пространство с паспортом. Владелец списка коллекций и их конфигурации эмбеддингов — **чертёж**, секция `collections` (40). Здесь — механика.

### Жизненный цикл коллекции и паспорта

Цикл разомкнут явно: **конфиг чертежа — источник, паспорт — отпечаток, сверка — следствие.**

1. **Создание (инжест).** `make seed` читает `collections` из чертежа: создаёт коллекцию под размерность модели (размерность узнаётся пробным вызовом эмбеддера, не хардкодом) и кладёт **паспорт** — служебную точку с зарезервированным id `0`:
   `{provider, model, dim, created_at, source}`. Точка-паспорт работает на любой версии Qdrant — произвольных метаданных коллекции у Qdrant нет, и мы на них не опираемся.
2. **Сверка (старт агента).** `self.storage.qdrant(collection)` читает паспорт и сравнивает с блоком `collections` чертежа. Расхождение — громкий отказ: *«коллекция regulations проиндексирована Embeddings/gigachat (1024), в чертеже bge-m3 — исправь чертёж или переиндексируй»*. Вектора разных моделей несравнимы; тихий мусорный поиск запрещён законом fail-fast.
3. **Переиндексация.** Смена модели эмбеддингов = `make seed` заново: коллекция пересоздаётся целиком, паспорт обновляется. Других путей нет.

### Инжест

```
make seed [COLLECTION=regulations|incidents|all]     # по умолчанию all; идемпотентен — пересоздаёт целиком
knowledge/<коллекция>/*.md → чанки (по заголовкам, ~500 токенов, перехлёст 50)
                          → вектора (модель из чертежа)
                          → Qdrant c payload {source, heading, text} + паспорт id=0
```

Раскладка `knowledge/<коллекция>/` — владелец принадлежности файлов (55). Утилита `tools/ingest` предметки не знает. Поиск исключает точку id=0.

`make seed` встроен в `make up` — чистый клон получает готовые знания без отдельной команды (60: golden path).

## Чего на складах нет — по принципам

- Конверты в базах не хранятся: шина — не склад, склад — не шина.
- Кэшей «для скорости» нет: не проблема этого масштаба.
- Доступ агентов к складам мимо SDK запрещён — проверки паспорта живут в одном месте.
