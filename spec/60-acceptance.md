Каждый пункт — проверка руками. Система принята, когда все строки — да.

## Требования задания → проверка

| № | Требование | Как проверить |
|---|---|---|
| 1 | Единый протокол взаимодействия | все сообщения — конверты одной схемы: видно в RabbitMQ UI и в `core/` (одна Pydantic-модель); агент не может собрать конверт мимо SDK |
| 2 | Агенты — независимые сервисы, каждый в своей среде | `docker ps` — по контейнеру на экземпляр; `docker exec agentarium-core pip list` — в ядре нет LangChain; в образах агентов — есть |
| 3 | Добавление агента не требует изменения ядра | конфигурация B добавляет auditor: `git diff dba-base dba-extended` — правки только в YAML; новый тип агента = новый модуль через SDK, `core/` нетронут |
| 4 | Сквозная наблюдаемость | Jaeger: один trace_id от `POST /requests` до результата, спаны всех агентов; логи JSON с trace_id |
| 5 | Общие хранилища по потребности | Postgres и Qdrant — сервисы compose; агенты берут клиентов у SDK; knowledge и auditor делят один Qdrant с разными коллекциями |
| 6 | Отказоустойчивость не в агентах | `docker kill` любого агента посреди `make demo` → система жива, конверт в очереди, restart доделывает; ретраи и `task.failed` — в SDK, в коде агентов ни строчки про это |
| 7 | Контейнеризация, запуск одной командой | `cp env.example .env && make up` на чистой машине с docker |

## Acceptance-критерии задания

| Критерий | Проверка |
|---|---|
| Конфигурация через YAML/env, не хардкод | состав, связи, модели, коллекции — в `configs/*.yaml`; секреты и адреса — env; в коде агентов нет имён других агентов |
| Падение одного агента не блокирует (healthcheck/restart) | сценарий-доказательство из `50-observability.md`, входит в демо |
| Пример 2+ агентов: парсер + RAG + исполнитель — рабочий | `make demo`: три заявки проходят цепочку, ответ содержит план и результаты реальных проверок против target-db |

## Наши критерии сверх задания

| Критерий | Проверка |
|---|---|
| Гибкость доказана двумя конфигурациями | `make demo CONFIG=dba-extended` — auditor добавлен только YAML-ом |
| Fail-fast эмбеддингов | поменять модель эмбеддингов в конфиге knowledge → старт падает с человеческим сообщением про паспорт коллекции |
| Fail-fast чертежа | маршрут в несуществующий экземпляр → система не поднимается, список дыр в сообщении |
| Невалидная заявка наблюдаема | заявка №3: отказ виден в трейсе как нормальный путь |
| Чистый клон | финальная проверка G3: `git clone` во временную папку, `.env` из примера, `make up && make demo` — от нуля до демо без правок |
