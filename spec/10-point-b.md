## Точка А

Мультиагентные системы собираются как монолиты: пайплайн зашит в код одного приложения, агенты знают друг друга по именам, добавление узла — правка ядра, падение одного узла кладёт всё, путь запроса восстанавливается по логам вручную.

## Точка Б

Существует платформа, на которой мультиагентная система **собирается из конфига, а не пишется заново**:

1. Состав и связи системы описаны YAML-файлом — чертёж графа: агенты-узлы, маршруты-стрелки.
2. Агенты — независимые сервисы в своих контейнерах со своими зависимостями. Они не знают друг о друге: закончил работу — отдал конверт в шину, кто следующий — решает маршрут.
3. Новый тип агента пишется через SDK как одна функция `handle()` и подключается **без изменения ядра**.
4. Путь любой заявки виден насквозь: один trace_id от входа до ответа, каждый шаг каждого агента — в трейсе.
5. Падение агента не блокирует систему: конверт доживает в очереди, агент рестартует и доделывает.
6. Общие склады (состояние, знания) доступны агентам по конфигу.
7. Всё поднимается одной командой.

## Сцена приёмки — глазами проверяющего

```
git clone … && cp env.example .env && make up      # одна команда: чертёж → система, знания засеяны
make demo                                           # заявки проходят парсер → RAG → исполнитель,
                                                    # в ответе план работ и проверки против живой БД
открыл Jaeger        → путь заявки через агентов, водопадом
открыл RabbitMQ UI   → очереди агентов, конверты в полёте
make demo CONFIG=dba-extended                       # система пересобралась: новый агент-аудитор,
                                                    # ядро и старые агенты не тронуты
make demo & demo/chaos.sh knowledge                 # авария агента ровно посреди обработки:
                                                    # система жива, конверт дождался, restart доделал
открыл core/         → предметки и LangChain там нет
```

Если каждая строка сцены — правда, точка Б взята. Формальная проверка — `60-acceptance.md`.
