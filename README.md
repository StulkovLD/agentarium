# agentarium

**Платформа для сборки распределённых мультиагентных систем.** Агенты — независимые сервисы в контейнерах, которые не знают друг о друге. Система собирается из YAML-чертежа: агенты — узлы, маршруты — стрелки, шина исполняет граф. Добавление агента любого типа не требует изменения ядра.

> Текущая фаза, версия и гейты — единственный источник: `PROGRAM_STATUS.yaml`. Архитектура полностью написана на бумаге (`spec/`) до кода и прошла четыре раунда адверсариального кросс-модельного ревью; имплементация идёт по `spec/70-roadmap.md`.

Четыре вещи, которые стоит увидеть за первые две минуты:

- **Оркестрация без оркестратора.** Процесса-роутера нет: маршруты из YAML становятся биндингами RabbitMQ, доставляет сам брокер. Шина тупая, интеллект по краям, связность — декларация.
- **LLM не пишет SQL.** Исполнитель выбирает из allowlist типизированных read-only проверок под отдельной ролью с `pg_monitor` и `statement_timeout` — инъекция в тексте заявки упирается в стену: сырого SQL от модели не существует по построению.
- **Каждый агент — своя модель.** Модель — строчка чертежа: в PoC parser и knowledge берут лёгкую GigaChat-2, а исполнитель — GigaChat-3-Ultra, потому что планирование и выбор проверок тяжелее классификации. Сменить модель или провайдера (GigaChat ↔ локальная bge-m3 для эмбеддингов) — правка YAML и переиндексация, код не трогается.
- **Отказ — первоклассный сценарий, не оговорка.** Агента убивают посреди обработки — конверт доживает в quorum-очереди, рестарт доделывает; ядовитый конверт после трёх доставок уходит в dlq и честно завершает заявку как `failed`. Путь аварии нарисован в `spec/diagrams/failure-path.mmd`, механизм закрыт интеграционными тестами (`tests/integration/test_sdk.py`).

## Проверяющему: три команды без ключа и без Docker

Чтобы убедиться, что репозиторий живой, ключ GigaChat не нужен — нужен только [uv](https://docs.astral.sh/uv/):

```bash
uv sync --group dev      # поставить окружение
make lint                # ruff по всему коду
make test                # 110 юнит-тестов, ~8 секунд, без единого внешнего сервиса
```

Всё зелёное из чистого клона. Логика ядра (протокол, доставка, fail-fast, allowlist проверок) покрыта юнит-тестами и читается без запуска.

## Запуск живьём

```bash
cp env.example .env      # вписать ключ GigaChat (GIGACHAT_CREDENTIALS)
make demo                # одной командой: поднять систему, засеять знания, прогнать заявки
```

Нужны Docker (≥4 ГБ памяти — локальная модель эмбеддингов bge-m3 живёт в Ollama-контейнере), [uv](https://docs.astral.sh/uv/) и ключ GigaChat. Первый прогон дольше остальных: собираются образы агентов и подтягивается bge-m3 (~1.2 ГБ) — заложи 3–5 минут. `make demo` поднимает 10 контейнеров (шина, склады, наблюдаемость, целевая БД, локальный эмбеддер, агенты, шлюз), индексирует регламенты и прогоняет три заявки. Наружу платформа не занимает дефолтные порты сервисов (Postgres — 5433, Ollama — 11534): у разработчиков на 5432/11434 часто уже что-то слушает.

Что видно в конце `make demo` (реальный прогон, конфигурация A):

```
Заявка: Проверь, какие доступы у пользователя ivanov в базе billing на проде
  путь:    parser → knowledge → executor   (интент: check_access)
  проверка user_roles({'user': 'ivanov'}) → [{'role': 'billing_rw'}]   ← из живой БД
  проверка user_privileges({'user': 'ivanov', 'database': 'billing'}) → []
  источники: [access-provisioning.md, role-model.md]
  ВЕРДИКТ:  ivanov состоит в billing_rw, прямых привилегий на базу нет. План и факты согласуются.

Заявка: Что за ерунда:))  → отказ: не является заявкой DBA
```

Заявка человеческим языком превращается в план работ с **реальными read-only проверками против живой целевой БД**. Путь заявки виден в Jaeger (`http://localhost:16686`), очереди — в RabbitMQ UI (`http://localhost:15672`), подать свою заявку из браузера — Swagger шлюза (`http://localhost:8000/docs`).

## Что внутри

```
Проверяющий ──POST /requests──▶ gateway ──▶ [шина: RabbitMQ] ──▶ parser ──▶ knowledge ──▶ executor
                                   │                                (LLM)      (RAG)        │
                                   ◀──────────────── plan.ready ◀───────────────────────────┘
             склады: PostgreSQL (состояние) · Qdrant (знания) · глаза: Jaeger (трейсы)
```

- **Ядро** (`core/`): конверт-протокол, шина с очередями, SDK агента, декларативные маршруты, наблюдаемость, отказоустойчивость. Предметки и LLM-фреймворков в ядре нет — проверяется списком зависимостей контейнера.
- **PoC** (`agents/`, `configs/`): система обработки заявок DBA — «проверь доступы ivanov на проде» превращается в план работ с реальными проверками против живой целевой БД. Мозги агентов — LangGraph + GigaChat, эмбеддинги для RAG — локальная bge-m3 через Ollama.
- **Гибкость доказывается демо:** вторая конфигурация (`configs/dba-extended.yaml`) добавляет агента-аудитора — новый тип подключается страницей кода через SDK, а маршруты и финал переключаются **только чертежом**; ядро и существующие агенты не тронуты. Падение любого агента посреди демо систему не останавливает.

> **Что прогнано живьём, а что нет** (честно, `PROGRAM_STATUS.yaml` — источник истины). Конфигурация A прогнана на живом GigaChat: три заявки, реальные проверки БД (вывод выше). Живучесть доказана интеграционными тестами (`tests/integration/test_sdk.py`: конверт переживает смерть консюмера, ядовитый конверт → dlq, вотчдог) и наблюдалась в бою. Конфигурация B (аудитор) и наглядная chaos-запись — код готов и покрыт тестами, живой прогон отложен: бесплатный лимит токенов GigaChat конечен (эмбеддинги вынесены на локальную bge-m3 именно поэтому). Гейт G3 (полный `make demo` из чистого клона) держит тот же лимит — не код.

## Как это построено

Разработка ведётся spec-first: система полностью спроектирована на бумаге до кода — принципы, контракты, точка Б, план стройки слайсами с гейтами (`spec/`). Имплементация — AI-assisted: агентная разработка по принятой спеке, план-факт один к одному, каждый слайс проходит тесты и ревью. Правила процесса — `CLAUDE.md`.
