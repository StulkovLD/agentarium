# agentarium

[![License: MIT](https://img.shields.io/badge/License-MIT-green.svg)](LICENSE)

**Платформа для сборки распределённых мультиагентных систем.** Агенты — независимые сервисы в контейнерах, каждый обрабатывает свой тип сообщений и не знает о существовании остальных. Систему описывает один YAML-файл: перечень агентов и маршруты между ними. Добавить агента нового типа можно без правки ядра платформы.

Понять суть — минута чтения. Убедиться, что всё работает, — три команды, без ключей и без Docker (ниже).

---

## Что это

Ядро платформы (`core/`) — это шина сообщений, протокол, SDK агента и маршрутизация из конфигурации. Ядро не содержит предметной логики и не привязано к LLM-фреймворку.

Поверх ядра собран рабочий пример (`agents/`, `configs/`): система обработки заявок администратора баз данных. Заявка на естественном языке («проверь доступы ivanov на проде») превращается в план работ с реальными read-only проверками против целевой БД.

## Проверка без ключа и без Docker

Нужны [uv](https://docs.astral.sh/uv/) и `make` (Python 3.12 uv поставит сам):

```bash
uv sync --group dev      # поставить окружение
make lint                # ruff по коду — без замечаний
make test                # юнит-тесты — без внешних сервисов
```

Ожидаемый результат `make test`:

```
111 passed, 4 skipped, 38 deselected
```

111 юнит-тестов проходят за секунды из чистого клона. 4 пропущенных — e2e-тесты на живом GigaChat (`make test-e2e`); 38 отфильтрованных — интеграционные, им нужны сервис-контейнеры (`make test-integration`). Логика ядра — протокол, доставка, fail-fast, allowlist проверок — покрыта юнит-тестами и читается без запуска системы.

## Запуск живьём

```bash
cp env.example .env      # вписать ключ GigaChat в GIGACHAT_CREDENTIALS
make demo                # поднять систему, проиндексировать знания, прогнать заявки
make down                # остановить и снести стенд, когда закончите
```

**Требуется:** Docker, [uv](https://docs.astral.sh/uv/), `make` и ключ GigaChat с доступом к моделям чата и эмбеддингам (у корпоративных ключей есть по умолчанию). Для корпоративного ключа в `.env` поставьте `GIGACHAT_SCOPE=GIGACHAT_API_CORP` (по умолчанию там персональный `GIGACHAT_API_PERS`).

**Сколько ждать:** первый прогон дольше остальных — собираются образы агентов. Обычно 1–2 минуты.

**Что делает `make demo`:** поднимает 9 контейнеров (шина, хранилища, трейсинг, целевая БД, три агента и шлюз), индексирует регламенты и прогоняет три заявки. И чат, и эмбеддинги для RAG идут по API GigaChat — локальных моделей на этом пути нет.

**Нет доступа к эмбеддингам GigaChat?** Есть офлайн-путь на локальной модели bge-m3 (её тянет отдельный Ollama-контейнер): `make demo CONFIG=dba-base-local`. Дольше (модель ~1.2 ГБ качается при первом старте — 5–15 минут в зависимости от сети, Docker с ≥4 ГБ памяти), но RAG работает без обращения к API за векторами. Это же — живая демонстрация того, что провайдер эмбеддингов переключается одной строкой конфигурации.

Пароли в репозитории (`agentarium/agentarium`, роли демо-БД) — публичные значения локального стенда, не секреты. Настоящий ключ GigaChat лежит только в `.env`, которого в git нет.

Сервисы, чей дефолтный порт разработчик часто уже занял своим, сдвинуты наружу: Postgres — на 5433, Ollama (на офлайн-пути) — на 11534. Остальные на штатных портах (RabbitMQ 5672, Qdrant 6333, Jaeger 16686).

**Пример вывода** (конфигурация A):

```
Заявка: Проверь, какие доступы у пользователя ivanov в базе billing на проде
  путь:    parser → knowledge → executor   (интент: check_access)
  проверка user_roles({'user': 'ivanov'}) → [{'role': 'billing_rw'}]   ← из живой БД
  проверка user_privileges({'user': 'ivanov', 'database': 'billing'}) → []
  источники: [access-provisioning.md, role-model.md]
  ВЕРДИКТ:  ivanov состоит в billing_rw, прямых привилегий на базу нет. План и факты согласуются.

Заявка: Что за ерунда:))  → отказ: не является заявкой DBA
```

Посмотреть систему изнутри:

| Окно | Адрес | Что видно |
|---|---|---|
| Swagger шлюза | http://localhost:8000/docs | подать свою заявку из браузера |
| Jaeger | http://localhost:16686 | путь заявки через агентов как водопад трейсов |
| RabbitMQ UI | http://localhost:15672 | очереди и сообщения в полёте |

## Как устроено

```
Клиент ──POST /requests──▶ gateway ──▶ [RabbitMQ] ──▶ parser ──▶ knowledge ──▶ executor
                             │                         (LLM)      (RAG)          │
                             ◀────────────── plan.ready ◀────────────────────────┘
      хранилища: PostgreSQL (состояние) · Qdrant (знания) · трейсы: Jaeger
```

Четыре ключевых решения:

- **Оркестрация без центрального оркестратора.** Отдельного процесса-маршрутизатора нет. Маршруты из конфигурации становятся биндингами RabbitMQ, доставку выполняет сам брокер. Агент публикует результат и не знает, кто следующий, — это решает таблица маршрутов.
- **LLM не пишет SQL.** Агент-исполнитель выбирает проверки из allowlist типизированных read-only функций (`user_roles`, `user_privileges`, `pg_version` и т.д.). К целевой БД он ходит под отдельной ролью с `pg_monitor`, `statement_timeout` и READ ONLY-транзакциями. Свободного SQL от модели в системе нет по построению — prompt-инъекция в тексте заявки упирается в это ограничение.
- **У каждого агента своя модель.** Модель — поле в конфигурации агента. В примере parser и knowledge используют лёгкую GigaChat-2, а executor — GigaChat-3-Ultra (планирование и выбор проверок тяжелее классификации). Смена модели или провайдера, включая переход на локальную bge-m3 для эмбеддингов, — это правка YAML, код не меняется.
- **Отказоустойчивость на механике брокера.** Агента можно убить посреди обработки: сообщение доживает в quorum-очереди, рестарт доделывает работу. «Ядовитое» сообщение после трёх доставок уходит в dead-letter-очередь, и заявка честно завершается статусом `failed`. Механизм закрыт интеграционными тестами на живом брокере (`tests/integration/test_sdk.py`).

Подробнее, в формате «решение → почему → что было бы без него», — в `docs/ARCHITECTURE.md`.

## Структура репозитория

| Папка | Что там |
|---|---|
| `core/` | ядро платформы: протокол сообщений (`Envelope`), шина, SDK агента, шлюз |
| `agents/` | типы агентов примера (`parser`, `rag`, `executor`, `auditor`) и каталог типов |
| `configs/` | конфигурации систем в YAML (`dba-base`, `dba-extended` и др.) |
| `spec/` | спецификации 00–70: архитектура и контракты |
| `knowledge/` | базы знаний для RAG (`regulations`, `incidents`) |
| `tests/` | юнит-, интеграционные и e2e-тесты |
| `demo/` | запуск демо (`run_demo.py`) и сценарий отказа (`chaos.sh`) |
| `tools/` | индексатор баз знаний в Qdrant |
| `infra/` | init-скрипты целевой БД |
| `docs/` | архитектура и гайд по сборке своих систем |

## Что работает и ограничения

- **Конфигурация A работает end-to-end на живом GigaChat**: заявка проходит весь конвейер (parser → knowledge → executor), executor выполняет реальные read-only проверки против целевой БД (пример вывода выше).
- **Отказоустойчивость** покрыта интеграционными тестами на живом брокере (`tests/integration/test_sdk.py`: сообщение переживает смерть консюмера, ядовитое сообщение уходит в dlq, вотчдог будит зависший обработчик).
- **Эмбеддинги по умолчанию** идут по API GigaChat — нужен ключ с доступом к эмбеддингам. Без него работает офлайн-путь на локальной bge-m3 (`make demo CONFIG=dba-base-local`).
- **Конфигурация B** (добавлен агент-аудитор) собрана и покрыта тестами; переключение на неё — `make demo CONFIG=dba-extended`.
