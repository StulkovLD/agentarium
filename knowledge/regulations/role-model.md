# Регламент ролевой модели PostgreSQL

## Назначение

Регламент фиксирует единую ролевую модель для инсталляций под сопровождением команды DBA.
Модель отделяет носителя привилегий (групповая роль) от субъекта (логин), делает права
воспроизводимыми и переносимыми между окружениями и упрощает аудит.

## Базовый принцип: группы носят права, логины входят в группы

- **Групповые роли** (`NOLOGIN`) — контейнеры привилегий по уровню доступа к набору объектов.
  Именно им выдаются гранты на таблицы, схемы, последовательности.
- **Логины** (`LOGIN`) — учётки людей и приложений. Собственных грантов на объекты у них в норме
  нет; права они получают членством в групповых ролях.
- Смена уровня доступа человека = изменение членства, а не переписывание десятков грантов.

## Стандартный набор групповых ролей на базу

Для базы `billing` (шаблон переносится на любую БД подстановкой префикса):

| Роль | Назначение | Права |
|---|---|---|
| `billing_ro` | только чтение | `CONNECT`, `USAGE` на схему, `SELECT` на таблицы |
| `billing_rw` | чтение и запись данных | `billing_ro` + `INSERT/UPDATE/DELETE`, `USAGE` на последовательности |
| `billing_owner` | владелец объектов, DDL | владеет таблицами схемы, право `CREATE` на схему |
| `billing_app` | сервисная роль приложения | обычно `billing_rw` + `CONNECTION LIMIT` |

`billing_rw` включает `billing_ro` через членство — иерархия ролей без дублирования грантов:

```sql
GRANT billing_ro TO billing_rw;
```

## Наследование: INHERIT против NOINHERIT

- Роли людей и приложений создаются с `INHERIT` (по умолчанию) — привилегии групп действуют
  автоматически, без `SET ROLE`.
- Автоматизации и сервисные роли, которым нужен явный контроль, могут создаваться `NOINHERIT` и
  повышать права через `SET ROLE` под конкретную операцию.
- Важное следствие для мониторинга: роль-исполнитель проверок (`readonly_executor`) создаётся с
  `INHERIT`, иначе членство в `pg_monitor` не даст своих привилегий без явного `SET ROLE`.

## Владение объектами

- Объекты владеет **`billing_owner`**, не человек. Тогда увольнение инженера не тянет за собой
  `REASSIGN OWNED`.
- Все миграции применяются от имени владельца (через `SET ROLE billing_owner` или отдельную
  учётку деплоя), чтобы вновь созданные объекты сразу принадлежали правильной роли.

## Права по умолчанию

Гранты `... ON ALL TABLES` покрывают только существующие объекты. Чтобы новые таблицы сразу были
доступны группам, задаём права по умолчанию от имени владельца:

```sql
ALTER DEFAULT PRIVILEGES FOR ROLE billing_owner IN SCHEMA public
    GRANT SELECT ON TABLES TO billing_ro;
ALTER DEFAULT PRIVILEGES FOR ROLE billing_owner IN SCHEMA public
    GRANT INSERT, UPDATE, DELETE ON TABLES TO billing_rw;
```

## Предопределённые роли PostgreSQL

Использовать штатные роли вместо суперпользователя, где хватает:

- `pg_monitor` — чтение статистики и настроек (включает `pg_read_all_stats`,
  `pg_read_all_settings`, `pg_stat_scan_tables`). Нужна ролям мониторинга, чтобы видеть
  `pg_stat_activity` шире собственной сессии.
- `pg_read_all_data` — глобальное чтение всех таблиц без индивидуальных грантов.
- `pg_signal_backend` — отмена/терминация чужих сессий без суперправ.

## Роль-исполнитель автоматических проверок

Отдельный, изолированный паттерн для систем, выполняющих проверки против базы:

```sql
CREATE ROLE readonly_executor LOGIN PASSWORD '***';
GRANT pg_monitor TO readonly_executor;                 -- видит активность всего кластера
GRANT CONNECT ON DATABASE billing TO readonly_executor;
GRANT USAGE ON SCHEMA public TO readonly_executor;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly_executor;
ALTER ROLE readonly_executor SET default_transaction_read_only = on;  -- любые записи отклоняются
ALTER ROLE readonly_executor SET statement_timeout = '5s';            -- ни один запрос не зависнет
```

Роль намеренно не входит в `billing_rw`/`billing_owner`: у автоматизации только чтение, жёсткий
таймаут и read-only транзакции на уровне самой роли — защита в глубину поверх отсутствия
write-грантов.

## Гигиена схемы public

На свежей базе снять неявные права:

```sql
REVOKE CREATE ON SCHEMA public FROM PUBLIC;
REVOKE ALL ON DATABASE billing FROM PUBLIC;
```

В PostgreSQL 15+ `CREATE` на `public` уже не выдаётся `PUBLIC` по умолчанию — но на
мигрированных с 14 и ниже базах это проверяется явно.

## Пароли и подключения

- `password_encryption = scram-sha-256`; метод `trust` в `pg_hba.conf` — только для локального
  сокета обслуживания, никогда для сетевых подключений.
- Сервисным ролям — `CONNECTION LIMIT`, чтобы одно приложение не выело пул целиком.
