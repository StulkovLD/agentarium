# Регламент обновления минорной версии PostgreSQL

## Назначение

Регламент описывает порядок планового обновления минорной версии PostgreSQL внутри одной
мажорной ветки (например, 16.1 → 16.3). Минорный релиз содержит только исправления
безопасности и ошибок; формат хранения данных на диске и системный каталог не меняются.
Поэтому обновление сводится к замене исполняемых файлов и перезапуску сервиса — ни `pg_upgrade`,
ни выгрузка/загрузка данных не требуются.

## Область применения

- Все продуктивные и предпродуктивные инсталляции PostgreSQL 13–16 под сопровождением команды DBA.
- Кластеры с потоковой физической репликацией (primary + один или несколько standby).
- Обновление в пределах одной мажорной версии. Переход между мажорными версиями — отдельный
  регламент «Обновление мажорной версии PostgreSQL».

## Предусловия

Прежде чем открывать окно обслуживания, дежурный DBA обязан подтвердить:

1. Прочитаны release notes целевого минора. В подавляющем большинстве случаев дополнительных
   ручных шагов нет, но иногда фикс требует пересборки индексов (`REINDEX`) или обновления
   расширения — такие исключения всегда указаны явно в разделе «Migration to version X.Y».
2. Есть свежая проверенная резервная копия и рабочий WAL-архив (см. регламент бэкапов). Точка
   восстановления не старше 24 часов.
3. Репликация здорова: `SELECT client_addr, state, replay_lag FROM pg_stat_replication;` — все
   standby в состоянии `streaming`, лаг близок к нулю.
4. Согласовано окно обслуживания. Для конфигурации с репликами простой сокращается до
   нескольких секунд за счёт переключения (switchover); без реплики простой равен времени
   перезапуска (обычно < 10 секунд плюс восстановление кэша).
5. Пакеты целевой версии доступны в корпоративном репозитории и совпадают по патч-уровню
   на всех узлах кластера.

## Порядок работ

### Кластер с репликами (рекомендуемый путь, минимальный простой)

1. Объявить начало окна, включить баннер режима обслуживания в приложении.
2. Обновить бинарники и перезапустить **сначала все standby** по очереди:
   `apt-get install postgresql-16=16.3-*` (или аналог пакетного менеджера) → `systemctl restart
   postgresql@16-main`. После каждого рестарта дождаться `streaming` и нулевого лага.
3. Выполнить контролируемое переключение на обновлённый standby:
   `pg_ctl promote` на реплике либо инструмент оркестрации (Patroni `switchover`). Клиенты
   переподключаются к новому primary.
4. Обновить бинарники и перезапустить бывший primary; вернуть его в кластер как standby.
5. При необходимости вернуть роль primary обратно (switchback) вне часа пик.

### Одиночный сервер

1. Объявить начало окна, дренировать соединения (остановить приём заявок на уровне приложения).
2. `apt-get install postgresql-16=16.3-*` → `systemctl restart postgresql@16-main`.
3. Дождаться готовности: `pg_isready`.

## Проверка после обновления

- `SELECT version();` — подтверждён целевой патч-уровень на каждом узле.
- В логе нет ошибок старта, есть строка `database system is ready to accept connections`.
- Смоук-запросы приложения проходят; ключевые дашборды зелёные.
- Репликация возобновилась: `pg_stat_replication` показывает `streaming`, лаг падает к нулю.
- Если release notes требовали `REINDEX` или `ALTER EXTENSION ... UPDATE` — шаги выполнены и
  зафиксированы в тикете.

## Откат

Минорные версии в пределах мажора двоично совместимы, каталог и файлы данных не тронуты. Откат =
установка предыдущего пакета и перезапуск. Восстановление из резервной копии для минорного
отката не нужно и не выполняется — это признак того, что что-то пошло сильно не так, и тогда
работает регламент аварийного восстановления, а не откат.

## Риски и пороги

- **Пропущенный ручной шаг из release notes.** Митигируется обязательным чтением раздела
  «Migration» до начала работ.
- **Рестарт под нагрузкой.** Дренировать соединения; `statement_timeout` и
  `idle_in_transaction_session_timeout` на клиентских ролях не дают зависшим транзакциям
  задержать остановку.
- **Рассинхрон патч-уровней узлов.** Primary и standby держим на одном патч-уровне; расхождение
  допускается только на время самого rolling-обновления.
- Плановая частота: применять минорные обновления в течение 30 дней после релиза; исправления
  критических CVE — во внеочередном окне в течение 7 дней.

## Коммуникации

Дежурный DBA публикует в канал изменений: планируемое окно, затрагиваемые инсталляции, ожидаемый
простой, контакт на время работ и отметку о завершении с результатом `SELECT version()`.
