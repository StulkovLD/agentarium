# Compliance-чеклист PostgreSQL

## Назначение

Периодический чеклист соответствия для инсталляций PostgreSQL под сопровождением команды DBA.
Проходится при ежеквартальном ревью и перед вводом новой инсталляции в продуктив. Каждый пункт —
проверяемый факт с командой сверки, а не «вроде настроено».

## Доступы и аутентификация

- [ ] Нет постоянных суперпользователей-людей.
      `SELECT rolname FROM pg_roles WHERE rolsuper AND rolcanlogin;` — только сервисный `postgres`.
- [ ] Пароли только SCRAM: `SHOW password_encryption;` = `scram-sha-256`.
- [ ] В `pg_hba.conf` нет `trust`/`password`/`md5` для сетевых подключений; сеть — только
      `scram-sha-256`, желательно поверх TLS (`hostssl`).
- [ ] Нет общих/разделяемых учёток; каждый логин именной.
- [ ] Временные доступы имеют `VALID UNTIL`; истёкшие роли вычищены.
- [ ] Проведено квартальное ревью прав, список подписан владельцами ресурсов.

## Шифрование

- [ ] Трафик шифрован: `SHOW ssl;` = `on`; сертификаты не просрочены.
- [ ] Данные на диске защищены (шифрование тома/ФС или TDE-решение) согласно классу данных.
- [ ] Резервные копии шифруются в объектном хранилище.

## Аудит и журналирование

- [ ] `log_connections` и `log_disconnections` включены.
- [ ] `log_statement = 'ddl'` минимум; для чувствительных БД — `pgaudit` с политикой по ролям.
- [ ] `log_line_prefix` содержит время, пользователя, БД, `application_name` и адрес.
- [ ] Логи уходят в централизованный сборщик, ретенция соответствует политике (обычно ≥ 1 год).

## Резервное копирование и восстановление

- [ ] Настроено непрерывное архивирование WAL; `SELECT * FROM pg_stat_archiver;` без роста
      `failed_count`.
- [ ] Есть свежая базовая копия в пределах RPO.
- [ ] Проведён тестовый восстановительный прогон за последний месяц, время уложилось в RTO.
- [ ] Ретенция и офсайт-хранение соответствуют политике (см. регламент бэкапов).

## Патч-менеджмент

- [ ] Минорные обновления применяются в течение 30 дней после релиза.
- [ ] Критические CVE закрываются во внеочередном окне в течение 7 дней.
- [ ] Версия в поддержке сообщества (не EOL); план перехода с EOL-версии зафиксирован.

## Управление изменениями

- [ ] Весь DDL идёт через ревьюируемые миграции, не руками на проде.
- [ ] Объекты владеет роль-владелец, не человек (см. регламент ролевой модели).
- [ ] Прод-изменения выполняются в согласованном окне с планом отката.

## Наблюдаемость и устойчивость

- [ ] Мониторятся: лаг репликации, свободное место (особенно `pg_wal`), длинные транзакции,
      idle-in-transaction, приближение wraparound (`age(datfrozenxid)`).
- [ ] Настроены алерты на архивирование WAL, отставание реплики и заполнение диска.
- [ ] `statement_timeout` и `idle_in_transaction_session_timeout` заданы для клиентских ролей.
- [ ] `autovacuum` включён и не отстаёт; нет таблиц с многомесячным `last_autovacuum`.

## Данные и приватность

- [ ] PII-столбцы классифицированы; при необходимости применяется Row-Level Security.
- [ ] Политика ретенции данных реализована (партиционирование/архивация/удаление).
- [ ] Маскирование/анонимизация для непрод-копий из продуктива.

## Разграничение обязанностей

- [ ] Запрос доступа и его согласование выполняют разные люди.
- [ ] Аварийный (break-glass) доступ протоколируется и разбирается постфактум.
- [ ] Тест аварийного восстановления (DR) проводится по расписанию, результат фиксируется.

## Итог ревью

По завершении чеклиста дежурный DBA формирует отчёт: дата, инсталляция, найденные отклонения,
план и сроки устранения. Незакрытые пункты переходят в тикеты с ответственными.
