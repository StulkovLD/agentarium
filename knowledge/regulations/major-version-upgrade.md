# Регламент обновления мажорной версии PostgreSQL

## Назначение

Регламент описывает переход между мажорными версиями PostgreSQL (например, 15 → 16). Мажорный
релиз меняет внутренний формат системного каталога и может менять формат хранения, поэтому
простой заменой бинарников обойтись нельзя. Требуется миграция данных одним из
санкционированных методов и обязательная проверка на копии продуктива до работ на проде.

## Область применения

- Плановый переход на новую мажорную версию по окончании срока поддержки текущей или ради новых
  возможностей.
- Инсталляции под сопровождением команды DBA, прошедшие валидацию на стенде.

## Выбор метода миграции

| Метод | Простой | Когда применять |
|---|---|---|
| `pg_upgrade --link` | минуты | БД большого объёма, окно ограничено; узел один, файлы на одной ФС |
| `pg_dump` / `pg_restore` | часы | небольшие БД (< 50 ГБ), нужна дефрагментация/смена кодировки |
| Логическая репликация | секунды переключения | требование почти нулевого простоя, есть время на подготовку |

По умолчанию для продуктива объёмом свыше 50 ГБ используется `pg_upgrade --link`; для критичных
систем без права на простой — логическая репликация с переключением в конце.

## Предусловия

1. Полная проверенная резервная копия снята непосредственно перед работами.
2. Прогон выполнен на стенде с копией продуктива продакшн-объёма; зафиксировано фактическое время.
3. Проверена совместимость расширений (`pg_stat_statements`, `postgis`, `pgcrypto` и пр.) с
   целевой версией; несовместимые версии обновлены в плане.
4. Проверены удалённые/устаревшие возможности целевой версии по release notes (изменения в
   параметрах, удалённые GUC, изменения поведения).
5. Установлены бинарники **новой** мажорной версии рядом со старой (разные каталоги `bin`/`data`).

## Порядок работ (pg_upgrade --link)

1. Объявить окно, остановить приём трафика, дренировать соединения.
2. Пробный прогон без изменений: `pg_upgrade --check` — проверяет совместимость каталога без
   модификации кластеров. Ошибки устранить до продолжения.
3. Остановить **оба** кластера (старый и новый). Работать под сервисной учёткой `postgres`.
4. Выполнить `pg_upgrade` с `--link` (жёсткие ссылки вместо копирования — быстро и без двойного
   места на диске). После `--link` старый кластер использовать нельзя.
5. Запустить новый кластер, обновить расширения: `ALTER EXTENSION <name> UPDATE;` по списку из
   `\dx`.
6. Пересобрать статистику планировщика — она не переносится: `vacuumdb --all --analyze-in-stages`.
   Без свежей статистики планы деградируют, это частая причина «после апгрейда всё медленно».
7. При смене ОС/версии glibc параллельно с апгрейдом — пересобрать индексы по текстовым столбцам:
   порядок сортировки collation мог измениться, иначе индексы становятся некорректными.
   `REINDEX` таргетно либо перевод на collation-провайдер ICU.

## Проверка после обновления

- `SELECT version();` — целевая мажорная версия.
- `\dx` — все расширения обновлены до версий, совместимых с новым мажором.
- Смоук приложения и ключевые запросы; сравнение планов критичных запросов с эталоном со стенда.
- Мониторинг ошибок и времени отклика в течение первых суток повышенного внимания.

## Откат

- Старый кластер после `--link` непригоден — план отката строится на резервной копии,
  снятой перед работами, либо на заранее поднятой параллельной старой инсталляции.
- Скрипт `delete_old_cluster.sh`, сгенерированный `pg_upgrade`, запускать **только** после
  подтверждённой валидации на проде (не раньше 24–48 часов стабильной работы).

## Риски и пороги

- **Деградация планов из-за отсутствия статистики** — обязательный `analyze-in-stages` сразу
  после старта, до открытия трафика.
- **Порча индексов из-за смены collation** — контролируется пунктом про `REINDEX`/ICU;
  это класс инцидента, уже случавшегося (см. историю инцидентов).
- **Несовместимое расширение** — ловится на стенде и `pg_upgrade --check`, не на проде.
- **Недооценка окна** — время берём из фактического прогона на стенде, а не из ожиданий.
- RPO: копия перед работами; RTO: не превышает согласованного окна плюс время отката.

## Коммуникации

Развёрнутый план-график с окном, методом, ответственными и точкой принятия решения о
завершении/откате согласуется с владельцем системы за неделю. Итог фиксируется в тикете.
