# Инцидент: смена collation при мажорном обновлении испортила уникальные индексы

## Карточка

- **Дата:** 22 марта 2025
- **Система:** db-billing (обновление PostgreSQL 15 → 16 с переездом на новую ОС)
- **Критичность:** P1 — некорректные результаты запросов, риск нарушения уникальности
- **Затронуто:** справочники клиентов и тарифов (текстовые ключи)

## Что произошло

После планового мажорного обновления, совмещённого с переездом на новый образ ОС, часть запросов
с фильтром и сортировкой по текстовым столбцам начала возвращать неверные результаты, а попытка
`REINDEX` уникального индекса упала с ошибкой дубликатов — хотя логически дубликатов не было.

## Хронология

- **Ночь работ** — выполнен `pg_upgrade --link` 15 → 16. Смуоки прошли, трафик открыли.
- **+1 день** — аналитики заметили, что выборки по текстовым ключам пропускают строки и
  сортируются «не по алфавиту».
- **+1 день** — DBA обнаружил, что образ новой ОС принёс другую версию glibc, а с ней —
  изменённый порядок сортировки для текущей locale. Индексы, построенные при старом порядке,
  стали неконсистентны новому.

## Причина

Индексы по текстовым столбцам физически упорядочены согласно collation. Порядок сортировки в
glibc-локали не является стабильным между версиями библиотеки: обновление ОS изменило порядок,
и B-tree индексы, построенные до апгрейда, перестали соответствовать новому сравнению. Планировщик
использовал «сломанный» индекс — отсюда пропущенные строки и ложные дубликаты при `REINDEX`.
Корень — обновление ОS/glibc проведено одновременно с апгрейдом БД без обязательного
пересоздания текстовых индексов.

## Как чинили

1. Идентифицировали все индексы по текстовым столбцам с collation по умолчанию
   (через `pg_index`/`pg_collation`).
2. Пересобрали их: `REINDEX INDEX CONCURRENTLY` по списку, ключевые уникальные — в первую очередь.
3. Проверили уникальность данных перед пересборкой уникальных индексов, чтобы отделить
   collation-артефакт от реальных дублей.
4. Сверили результаты контрольных запросов с эталоном до апгрейда.

## Чему научились

- В регламент мажорного обновления внесён обязательный шаг: при любом изменении версии
  ОС/glibc параллельно с апгрейдом — пересборка индексов по текстовым столбцам сразу после
  переключения, до открытия трафика.
- Для новых баз выбран collation-провайдер ICU с версионированием: он стабильнее к обновлениям ОС
  и явно сигнализирует о рассинхроне версии collation.
- Разнесли по возможности две операции: обновление БД и переезд ОС делаются отдельными окнами,
  чтобы источник изменений сортировки был однозначен.
- Добавлена проверка версий collation (`pg_collation` против ОС) в чеклист после апгрейда.
